## 0. TL;DR

1. **Git 面板 = 第三个受控面板**,挂在右 dock stack 的「文件 / Git / 已打开」之间,沿用 ADR-016 的 Panel Registry。它是**只读变更浏览器**:列当前仓库的已修改文件,点文件 → 跳到「已打开」面板看 diff。**不做** stage / commit / push / branch / log / stash 等 Git 仓库管理操作。
2. **文件条目统一抽象**:抽 `<FileListRow>`(`file-tree` / `git` / `file-panel` 共用同一渲染壳),承载图标 + 名称 + 状态标记 + hover/选中态。**左键行为由所在面板注入**(file-tree=展开目录/打开预览、git=打开 diff、file-panel=切 active tab),**右键菜单统一走现有 `ContextMenu`**,菜单项由面板按条目上下文动态构建。
3. **diff 渲染复用 FilePanel 路径**:diff 作为 `OpenedFile` 的新来源(虚拟文件 `marina-diff://<repo>/<path>`),由新增的 `GitService` 产出 unified diff 文本,交给既有 `TextViewer` 渲染。**不引入 diff 库作为新依赖**(见 §10)。

## 1. 背景与动机

### 1.1 当前状态

- 主工作区已落地 ADR-016 Panel System:`PanelRegistry` + `LayoutNode` + `LayoutHost`,右 dock 当前有 `file-tree`(`trigger: 'product-rule'`)+ `file-panel`(`trigger: 'program-push'`)两个面板。
- `FileTreeService` 已确立了"session 绑定 + owner 校验 + realpath 根包含校验 + 只读 + 不递归"的安全模式(见 `src/main/file-tree-service.ts`)。
- `FilePanelService` 已确立了"已打开文件状态机 + Viewer 分发(text/markdown/image)+ 自动刷新"的预览模式。
- `ContextMenu` 抽象已全局可用(`src/renderer/components/ContextMenu.tsx`),支持 `label / hint / icon / checked / disabled / danger / divider / onSelect`,Sidebar / Tab / 终端区均已在用。
- 软件定义书 §13.2 列出"Project / Workspace / Repo 容器层级""内置文件编辑器"为永远不做;§14.6 画了"终端管理 vs 文件管理"边界,但留了"受限只读导航"例外(ADR-016)。

### 1.2 为什么做

- **开发者真实诉求**:终端跑 agent / 改代码后,想立刻"看刚才动了哪些文件、动了什么",目前必须切到外部 Git GUI 或终端里 `git diff`,割裂上下文。JetBrains 的 Git 面板之所以好用,正是因为它把"变更文件清单 + 一键看 diff"内联进了工作区。
- **架构已为此预留**:ADR-016 §7 明确写"用户主动 Git 浏览 / 仓库管理:架构 ✅、产品 ❌ 待单独产品决策"。本 PRD 就是这个"单独产品决策"。
- **受限例外已立先例**:`FileTreePanel` 用同样的"只读 + 复用既有预览 + 安全校验"模式,把文件浏览带进来却没变成 IDE。Git 面板走同一条路。

### 1.3 为什么是现在

v0.2.x 远程后端主线已稳定,Panel System 已验证可扩展,file-tree/file-panel 的安全与预览路径已成熟,新增第三个面板的边际成本很低。

## 2. 产品边界(关键:先画红线)

### 2.1 **做**(本次范围)

- 列出当前 session 的 `currentCwd` 所在 Git 仓库的**工作区未提交变更**(`git status --porcelain` 语义):Modified / Added / Deleted / Renamed / Untracked / Conflicted,按状态分组或着色。
- 单击变更文件 → 跳到「已打开」面板,以 diff 形式预览(unified diff,旧版/新版对照)。
- 文件条目(`file-tree` / `git` / `file-panel`)统一为 `<FileListRow>`,右键弹 `ContextMenu`,菜单项按面板与条目上下文动态生成。
- diff 预览随仓库变更自动刷新(`fs.watch` 仓库,或切回面板时主动 refresh)。

### 2.2 **不做**(本次范围外,V1.1+ 视反馈)

- Stage / unstage / commit / amend / push / pull / fetch / merge / rebase / stash / cherry-pick —— **任何写 .git 的操作**。
- Branch 切换、branch 列表、tag 管理。
- Commit history / `git log` / blame。
- Stash 列表。
- 任意 ref 间 diff(只做 `HEAD ↔ 工作区` / `index ↔ 工作区`)。
- Merge conflict 的三方合并 UI(只标红 conflicted,点进去看冲突标记文本)。
- 远程仓库(SSH session 的远端 git)—— 同 file-tree 的 SSH 策略:不支持就明确显示不支持,不引入远端 git 协议。

> 这些"不做"的理由见 §4 哲学对齐。它们**不是技术做不到**,是产品定位不碰。

### 2.3 **永远不做**(碰了等于变 IDE / Git GUI)

- 内置 commit 编辑器、commit message 模板。
- Branch 可视化图(gitk / GitKraken 式 DAG)。
- 任意 ref / 任意 commit 间 diff 浏览器。
- PR / MR 集成。
- `.gitignore` 编辑器。
- 把 Git 仓库提升为 Marina 的一等容器(`Repo` / `Project` 概念)—— 违反 §13.2。

## 3. 用户故事

### 3.1 主流程:改完代码看变更

> Alice 在 Marina 里跑 `claude code` 让 agent 重构了 `auth.ts`。她点右 dock 的「Git」tab,看到 `auth.ts` 标着橙色 M(Modified),下面还多了 `auth.test.ts` 标着绿色 A(Added)。她点 `auth.ts` → 面板自动切到「已打开」,右侧渲染出 unified diff,删了哪几行、加了哪几行一目了然。她看完关掉 diff tab,继续在终端里指挥 agent。

### 3.2 副流程:右键菜单统一

> Bob 在「Git」面板右键 `auth.ts` → 菜单弹出:「打开 diff」「在文件树中定位」「复制路径」「在 Explorer 中显示」。他选「在文件树中定位」,面板切到「文件」并自动展开滚动到 `auth.ts`。
>
> 同一个 Bob 在「文件」面板右键任意文件 → 菜单:「打开」「复制路径」「在 Explorer 中显示」。
>
> 同一个 Bob 在「已打开」面板右键一个 diff tab → 菜单:「关闭」「关闭其他」「关闭所有」「复制路径」。
>
> 三种场景的菜单长得一样、行为可预期,因为它们都来自同一个 `<FileListRow>` + `ContextMenu`。

### 3.3 边界流程:不是 Git 仓库

> Carol 的 session cwd 是 `D:\scripts`,不是 Git 仓库。她点「Git」tab → 面板显示空状态:"当前目录不是 Git 仓库"。不报错、不诱导她 `git init`。

### 3.4 边界流程:SSH session

> Dave 连了一个 SSH session,cwd 是 `/home/dave/proj`。**「Git」tab 不出现**(同 file-tree 的 SSH 拒绝,但不显示拒绝文案,直接不渲染)。

## 4. 哲学对齐(为什么这个范围合规)

| 哲学原则 | 本设计如何遵守 |
|---|---|
| 原则 1:UI 临时 | Git 面板是 session 临时视图,不持久化"上次看了哪个文件",切 session 即重挂(LayoutHost 已按 sessionId 重挂) |
| 原则 3:用户决策最少化 | 只读浏览,不弹"要不要 stage""commit message 填什么"等决策;diff 默认行为即点开就看 |
| §13.2 永远不做 | **不碰** Repo/Project 容器、不内置编辑器(只预览不编辑)、不碰 commit/push 等 .git 写操作 |
| §14.6 文件管理边界 | diff 是"只读查看",和 FileTreePanel 同属"受限只读导航例外";不提供文件修改、上传、任意路径浏览 |
| ADR-016 | 新面板注册进 PanelRegistry,不改 MainPane;LayoutNode 仍由 main 产品规则生成,renderer 不改树 |

**关键论证:为什么"看 diff"不违反"不做文件管理"?**

diff 预览本质是"把工作区与版本库的差异以文本形式呈现给用户看",与 FileTreePanel 的"把目录结构以列表呈现给用户看"同构 —— 都是**只读呈现**,不提供任何修改文件或修改仓库的 affordance。真正违反 §14.6 的是 commit/stage/push 这类**写操作**,本 PRD 明确不做。

## 5. 功能规格

### 5.1 Git 面板(新 PanelType:`git`)

#### 5.1.1 注册与位置

- 新增 `PanelType` id = `'git'`,`trigger: 'product-rule'`,`defaultDock: 'right'`。
- 右 dock stack 的 tab 顺序固定为 **`[文件] [Git] [已打开]`**(产品规则,renderer 不可改)。
- `LayoutNode` 由 main 端产品规则生成,在现有 `file-tree` 与 `file-panel` 之间插入 `git` leaf。
- width / collapsed 仍属于 `docks.right`(ADR-016 既有规则),切换 tab 不改几何。

#### 5.1.2 数据来源

- 新增后端服务 `GitService`(`src/main/git-service.ts`),仿 `FileTreeService` 模式:
  - 构造注入 `sessionLookup`(拿 `currentCwd` 与 `ownerWindowId`)+ `filePanelService`(打开 diff 复用预览路径)。
  - **每次请求**都校验 `ownerWindowId === requesterId`(防跨窗口读)。
  - **每次请求**都校验 `pathRefFromId(session.pathId).kind !== 'ssh'`(SSH 不支持,不引入远端 git)。
  - cwd 经 `fs.realpath` 后,向上查找 `.git` 目录(最多 N 层,见 §6.3 防穿越)确定仓库根。
- 数据通过现有 IPC 拉取,**不做乐观本地缓存**:renderer 每次激活面板 / 手动刷新时拉一次,仓库变更由 main 端 `fs.watch` 仓库根 + emit `evt:git:status-updated` 推送。

#### 5.1.3 状态展示

- **空状态**:cwd 不是 git 仓库 → 显示"当前目录不是 Git 仓库"(不报错、不诱导 init)。
- **SSH 不支持**:显示"SSH 会话不支持 Git 面板"(同 file-tree 的 SSH 拒绝模式)。
- **加载中**:首次拉取显示"正在读取变更…"(同 file-tree)。
- **无变更**:仓库干净 → 显示"工作区干净,无未提交变更"。
- **有变更**:按状态分组(可折叠,产品规则定死分组顺序):
  - 🔀 Conflicted(merge 冲突,置顶,红色)
  - M Modified(橙色)
  - A Added(绿色)
  - D Deleted(灰色)
  - R Renamed(蓝色,显示 `old → new`)
  - ? Untracked(灰色,可折叠,默认折叠以防 node_modules 污染)
- 每个文件条目:状态字母 + 文件图标 + 相对仓库根的路径 + hover 显示完整绝对路径 tooltip。
- 列表上限:**至多 500 个变更文件**(对齐 file-tree 的 `MAX_DIRECTORY_ENTRIES`),超出显示"仅显示前 500 项,请缩小变更范围"。

#### 5.1.4 交互

- **左键文件** → 调 `cmd:git:open-diff` → main 端 `GitService` 产出 diff → 走 `FilePanelService.openFile` 以虚拟 diff 文件形式进「已打开」→ emit `evt:file-panel:updated`(requestActivation=true)→ LayoutHost 自动切到「已打开」tab。这条路径与 file-tree 点击文件完全对称。
- **左键目录**(本 PRD 不涉及,Git 面板不展示目录树,只展示扁平变更文件列表)。
- **右键文件** → `ContextMenu`,见 §5.3。
- **手动刷新**:面板顶部无刷新按钮(违反"用户决策最少化"且 fs.watch 已自动刷新);但仓库变更检测失败时显示重试链接。
- **自动刷新**:main 端 `fs.watch` 仓库 `.git/index` 与工作区文件 → 防抖 300ms → 重算 status → emit。

#### 5.1.5 diff 内容

- unified diff 格式,默认 3 行上下文(对齐 `git diff` 默认)。
- 渲染走 `TextViewer`(diff 本质是文本),但加一层**轻量 diff 高亮**:新增行绿色背景、删除行红色背景、hunk header 灰底。这是 TextViewer 的新渲染模式,不改 FileViewer 的 kind 分发结构(见 §7.2)。
- 大文件 diff 截断:对齐 file-panel 的 `MAX_READ_TEXT_BYTES = 2MB`,超出显示"diff 过大已截断,请用外部工具查看完整差异"。

#### 5.2.3 右键菜单的统一规则

- 菜单项的 i18n 走现有 `useTranslation().tx(zh, en)`(中英双语,对齐 §6.6 现有约定)。
- 菜单关闭的焦点归还由现有 `ContextMenu` 的 `previousActiveElementRef` 机制处理(已在 CP-4 勘误里修过 FOC-5,复用即可)。

## 6. 技术设计

### 6.1 后端:新增 `GitService`(`src/main/git-service.ts`)

**职责**:为当前 owner session 提供只读 Git 状态查询与 diff 产出。

**不依赖第三方 Git 库**(见 §10),通过 `child_process.spawn` 调用系统 `git` 二进制(与 `node-pty` spawn 模式同构,但走 `child_process` 因为要拿 stdout 退出码,不要 PTY 语义)。

**核心方法**:

```typescript
class GitService extends EventEmitter {
  constructor(
    sessionLookup: GitSessionLookup,         // { get(id) -> { pathId, currentCwd, ownerWindowId } }
    filePanelService: FilePanelService,       // diff 产出后走 openFile
  );

  /** 返回仓库状态:null=不是仓库/SSH不支持;否则返回分组变更列表 */
  async getStatus(sessionId: string, requesterId: string): Promise<GitStatusSnapshot | null>;

  /** 产出某文件的 diff 并交给 FilePanelService 打开 */
  async openDiff(sessionId: string, requesterId: string, relativePath: string): Promise<FilePanelSnapshot>;

  /** session 销毁时清 watcher */
  onSessionDestroyed(sessionId: string): void;
}
```

**安全模式**(完全复刻 FileTreeService):
- 每次 `getStatus` / `openDiff` 都校验 `sessionLookup.get(sessionId).ownerWindowId === requesterId`。
- 校验 `pathRefFromId(session.pathId).kind !== 'ssh'`(SSH 拒绝)。
- `currentCwd` 经 `fs.realpath` canonicalize。
- `relativePath` 拒绝 `..` / 绝对路径 / NUL(同 file-tree 的 `resolveInsideRoot`)。
- diff 产出的目标文件必须 `isWithinRoot(repoRoot, realpath(target))`(同 file-tree 的根包含校验)。

**git 调用封装**:

```typescript
async function runGit(repoRoot: string, args: string[]): Promise<{ stdout: Buffer; stderr: string; exitCode: number }> {
  // spawn git -C <repoRoot> <args...>,限制 stdout 上限 8MB 防恶意大输出撑爆内存
  // git 二进制路径:settings.advanced.gitBinaryPath || 默认 PATH 查找
  // 超时 5s(防挂起的 hook)
}
```

**status 获取**:`git -C <root> status --porcelain=v2 -z --untracked-files=all`(v2 格式机器友好,字段稳定)。解析后分组返回。

**diff 产出**:`git -C <root> diff --no-color --full-index -- <relativePath>`(工作区 vs index);新增文件用 `git diff --no-color -- /dev/null <relativePath>` 或直接 cat;冲突文件用 `git diff --no-color --diff-filter=U`。

### 6.2 diff 作为 `OpenedFile` 的新来源

不引入新的 FileKind,而是用**虚拟路径**把 diff 接入既有 FilePanelService:

- `GitService.openDiff` 把 diff 文本写入 session 的 `MARINA_WORKSPACE` 下临时文件 `__marina_diff__/<sha>.diff`(受管目录,session 销毁自动回收),再调 `filePanelService.openFile(sessionId, thatPath)`。
- diff 文件的 `OpenedFile.kind` 经 `detectFileKind('.diff')` → 当前可能归为 `text` 或 `unknown`。**新增** `detectFileKind` 对 `.diff` 扩展返回新 kind `'diff'`(见 §6.4),触发 `DiffViewer` 渲染。
- 复用 `FilePanelService` 的全部既有能力:已打开列表、active tab、fs.watch 自动刷新、大小上限、关闭逻辑、requestActivation。

> **为什么用临时文件而不是新 FileKind + 新 IPC?** 临时文件方案让 diff 享受 FilePanelService **所有**既有机制(watcher 自动刷新当仓库再变、tab 管理、close),零改动既有状态机。代价是占一块受管磁盘(几 KB / diff),可接受。`MARINA_WORKSPACE` 已是受管临时目录,语义匹配。

### 6.3 仓库根查找与防穿越

```typescript
async function findRepoRoot(startReal: string): Promise<string | null> {
  // 从 startReal 向上查找 .git,最多 20 层(防 / 挂载点死循环)
  // 找到后 realpath(.git) 验证是目录或 gitlink 文件
  // 返回仓库根的 canonical path,找不到返回 null
}
```

`.git` 可能是 worktree 的 gitlink(文件内容 `gitdir: /path/to/main/.git/worktrees/x`),此时仍认作仓库,但 diff/stage 走主仓库。本 PRD 只读,gitlink 自动由 `git -C` 处理,不需手动解析。

## 7. 数据流与状态机

### 7.1 Git 面板状态机(renderer 视角)

```
mount/session 变化
  └─> cmd:git:get-status
       ├─ repoRoot=null, unavailable='not-a-repo' ─> 显示"不是 Git 仓库"
       ├─ unavailable='ssh-unsupported'           ─> 显示"SSH 不支持"
       ├─ unavailable='git-binary-missing'        ─> 显示"未找到 git,请在设置里指定路径" + 链接
       └─ 正常 ─> 渲染分组列表
            └─ 订阅 evt:git:status-updated ─> 重渲染
                 └─ 用户点文件 ─> cmd:git:open-diff
                      └─ FilePanelService.openFile(diff 临时文件)
                           └─ evt:file-panel:updated(requestActivation=true)
                                └─ LayoutHost 切到「已打开」tab
```

### 7.2 diff 文件生命周期

```
GitService.openDiff
  └─ runGit('diff', path) ─> diff 文本
       └─ 写入 MARINA_WORKSPACE/__marina_diff__/<sha>.diff
            └─ FilePanelService.openFile(thatPath)
                 ├─ detectFileKind('.diff') = 'diff'
                 ├─ 加入 files 列表,activePath = thatPath
                 ├─ ensureWatcher(.diff 文件)  ─> 文件变 = 仓库再变,但本 PRD 不自动重算 diff
                 └─ emit requestActivation=true
```

> **watcher 语义**:diff 临时文件被 GitService 写一次后不再变(fs.watch 不会触发),用户在 Git 面板再次点同一文件 → GitService 重算 diff → 覆写临时文件 → watcher 触发 → TextViewer 刷新。这条链天然实现"仓库变了再点一次就看新 diff"。

### 7.3 「在文件树中定位」程序化 API

新增 renderer 内的轻量 event bus(非 IPC,renderer 内部):

```typescript
// store 里新增 fileTreeFocus$ observable
fileTreeFocusRequest(sessionId, relativePath);  // 触发 file-tree 展开+滚动
```

`FileTreePanel` 订阅它,收到请求后逐级 `loadDirectory` 展开到目标文件并 `scrollIntoView`。这条路径要求 relativePath 能映射到 file-tree 的某个根(session-cwd 根下的相对路径)。若目标在 MARINA_WORKSPACE 根,同样处理;不在任何根内(理论不可能,因为 diff 目标必在 repoRoot 内,而 repoRoot ⊆ session-cwd 或其祖先 —— 见 §11 验收用例 6),则灰显该菜单项。

## 10. 依赖决策(边界 2:需开发者确认)

**问题**:diff 解析是否引入第三方库?

**候选**:
- (A)`parse-diff`(npm,~10k stars,纯 JS,解析 unified diff 为结构化对象)— 方便做行级高亮
- (B)`diff`(npm,jsdiff)— 功能更全但更重,且我们不需要"自己算 diff"(git 已算好)
- (C)**不引入**,自己写一个 ~80 行的 unified diff 行解析器(diff 格式极简,逐行扫 `+`/`-`/`@@` 即可)

**agent 建议**:**选 (C)**。理由:
- 我们只需"把 git 产出的 unified diff 文本按行着色",不需要计算 diff,parse-diff 的结构化能力用不上。
- 自写解析器 < 100 行,无依赖,无供应链风险,符合 AGENTS.md §1.2 边界 2"能不装就不装"。
- 若后续要做"行内 word diff"(同一行只高亮改动的词),那时再评估 parse-diff。

**需开发者裁决**:同意 (C) 自写?还是倾向 (A)?

> 若选 (A),走 AGENTS.md §1.2 边界 2 流程:在对话里说"想装 parse-diff,理由 X",等回复。

4. **diff 临时文件位置**:放 `MARINA_WORKSPACE/__marina_diff__/`(agent 建议)还是 `userData/git-diff-cache/<sessionId>/`?前者随 session 回收,后者需独立清理。

6. **GitService 是否复用 file-panel 的 HTTP 服务暴露给终端程序**?(让 agent 脚本主动推 "看这个 diff")—— 建议**本期不做**,先只做 UI 主动浏览,HTTP 暴露留 v0.4。

### 13.2 已识别的风险

| 风险 | 缓解 |
|---|---|
| 系统未装 git | 检测失败明确提示 + 设置项指定路径,不静默崩 |
| git 调用慢(大仓库) | 5s 超时 + status `--porcelain=v2` 已是最快路径;不递归、不 log |
| fs.watch 在网络盘不稳 | 降级:watch 失败不致命,面板激活时主动 refresh 兜底(同 file-panel 的 watch unavailable 降级) |
| diff 过大撑爆 | 2MB 截断(对齐 file-panel 既有上限) |
| 滑向 Git GUI | §2.3 永远不做清单 + ADR-017 白纸黑字焊死 |

### 13.3 不在本次范围(明确记进 BACKLOG,防"顺手做")

- diff 的 word-level 高亮(行内只标改动的词)
- diff 的 side-by-side 视图(左旧右新)
