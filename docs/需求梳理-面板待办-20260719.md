## 一、查看器（最影响「打开文件」体感）⭐ 高价值

### A1. TextViewer 语法高亮 ⭐⭐⭐（开发者反馈的「文本高亮」最可能指此）
- **现状**：打开任意代码文件（.ts/.py/.json/.yml/.sh…）都是纯黑白。DiffViewer 有高亮、打开文件本身反而没有 —— 体验割裂。
- **方案**：复用 DiffViewer 已有的 hljs 栈（`highlight.js/lib/core` + 按需注册 11 语言 + `EXT_TO_HLJS` 映射 + token→CSS 变量）。TextViewer 行级 `<div>` 上套 hljs 输出的 HTML 即可，与搜索的行内 `<mark>` 需协调（hljs span 嵌套 + mark 叠加）。
- **工作量**：中。hljs 栈已就位，主要是「hljs 输出 + 搜索 mark」的叠加渲染（DiffViewer 当时回避了这步，TextViewer 要正面解决）。
- **建议**：**做**。这是最明显的体感缺口。

### A2. DiffViewer 搜索行内字符高亮
- **现状**：在 diff 里 Ctrl+F，只跳到匹配行，匹配的字本身不高亮。
- **方案**：hljs 输出的 HTML span 里再定位 query 子串包 `<mark>`。复杂度来自 span 嵌套（C3 当时因此跳过）。
- **建议**：**次优先**。A1 做完 hljs+mark 叠加方案后，可顺带把 DiffViewer 的行内 mark 补上（同一套机制）。

### A3. DiffViewer 词级/字符级 diff（inline diff）
- **现状**：行级 diff（整行标红/绿），看不出「这一行里到底哪几个字改了」。
- **方案**：用 `diff-match-patch` 或 hljs token 对齐，在同一行内高亮改动的字符。
- **建议**：**未来**。价值高但工作量大，先排后面。

### A4. 行号槽
- **现状**：TextViewer / DiffViewer 都没有行号。
- **方案**：行级 `<div data-line>` 已就位，加一列固定宽行号 gutter 即可。
- **建议**：**做**，低成本。看代码/报错对行号是刚需。

### A5. DiffViewer 并排（split）视图
- **现状**：只支持 unified（上下堆叠）。
- **方案**：左右两列 old/new。toggle 切换。
- **建议**：**未来**。Git 只读浏览下 unified 已够用，split 是锦上添花。

### A6. 大文件保护
- **现状**：超大文件（如 10MB 日志）直接全量渲染行，会卡。
- **方案**：> N 行（如 50000）只渲染头部 + 「文件过大，已截断」提示，或上虚拟滚动。
- **建议**：**做**，防御性。现在 `file-truncated-mark` CSS 已有，逻辑没接。

## 二、菜单 / 交互 ⭐（开发者反馈点）

### B1. Git 树目录节点右键菜单 ⭐⭐⭐（开发者明确反馈）
- **现状**：Git 树里的文件夹右键完全无反应（FileListRow 没传 buildContextMenu）。
- **方案**：用统一生成器 `buildFileEntryMenu`，目录的能力集：
  - `primary`：展开/收起（同 file-tree 目录）
  - `relativePath`：复制相对路径（目录路径，贴 commit/import 实用）
  - `reveal`：在 Explorer 显示（resolve 目录绝对路径 → system:show）
  - 可选：「折叠所有」/「展开所有」？(能力接口未支持，需扩 `FileEntryContext`)
  - 不提供 `openFile`（目录无文件本体）/ `resolveAbsolutePath` 复制绝对路径（可提供，目录也有绝对路径）
- **工作量**：小。生成器 + 能力注入，半小时级。
- **建议**：**做**。这是统一抽象的应有之义——既然 file-tree 目录有菜单，git 目录也该有。

### B2. Git 树目录「复制绝对路径 / reveal」
- **现状**：无。
- **方案**：目录也走 `git:resolve-path`（已支持任意相对路径，目录也行）。
- **建议**：随 B1 一起做。

### B3. 目录「在此处打开终端」？
- **现状**：无。
- **方案**：在任意面板的目录右键加「在新 session 打开此目录」。
- **⚠️ 边界考量**：Marina 是「以路径为核心的终端管理器」，从目录反开 session 符合哲学。但要接 SessionManager.createSession(pathId?) —— 目前 createSession 需要 pathId（收藏/临时分类）。目录可能不在 path 树里，需决定归类（临时？）。
- **建议**：**讨论**。哲学 OK 但需定归类语义，可能引发 path 状态机讨论。先不动。

### B4. 文件节点「用默认程序打开」
- **现状**：无（只有「在 Explorer 显示」）。
- **方案**：`shell.openPath(absPath)`（非 reveal 的 showItemInFolder）。
- **建议**：**做**，小成本。.pdf/.png 双击用系统程序打开很自然。

### B5. 复制文件内容 / 复制 diff
- **现状**：无。
- **方案**：右键「复制文件内容」/「复制此文件 diff」。
- **建议**：**低优先**。长内容复制场景少。

### B6. 重新打开已关闭的文件（Recent / 最近关闭）
- **现状**：关闭 tab 后无迹。
- **方案**：file-panel 维护「最近关闭」列表，右键 dock 头部或菜单恢复。
- **建议**：**未来**。V1 非刚需。

## 三、搜索

### C1. 搜索结果列表（大纲式）
- **现状**：查看器内搜索只显示 `x/N` + 跳转，无「所有匹配的列表」。
- **方案**：侧边列出所有匹配行预览，点跳转。
- **建议**：**未来**。当前 x/N + Enter 翻页已可用。

### C2. 正则搜索
- **现状**：纯文本子串匹配。
- **方案**：搜索栏加 `.*` 切换，用 RegExp。
- **建议**：**未来**。高级用户需求。

### C3. 替换模式
- **现状**：无（只读）。
- **建议**：**永不做**。违反「面板只读」哲学（§13.2），文件编辑走外部编辑器。

### C4. 跨文件搜索（在仓库/目录里搜文本）
- **现状**：无。只能单文件搜。
- **方案**：「在 Git 变更里搜 X」「在此目录搜 X」。
- **建议**：**未来，值得讨论**。这是 VS Code 全局搜索的杀手锏，但工作量大（需 ripgrep 集成、结果聚合 UI）。可列为 V2 亮点。
