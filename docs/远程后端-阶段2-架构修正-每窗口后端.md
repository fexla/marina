# 远程后端 · 阶段2 架构修正:每窗口后端

> **状态:方案(待执行)** · 2026-07-05
> **触发**:用户测试视角发现"同时只能操纵一个后端,不能同时本地+远程"——
> 根因是阶段2 用了**全局 `activeRemoteProfileId`**,违背软件定义书原则4
> (窗口平等 / 窗口是观察工具,应用是守护进程)。

---

## 一、问题陈述

阶段2 的实现里,"连哪个后端"是**全局单值**:
- `RemoteDaemonProfilesFile.activeProfileId`(文件顶层一个字段)
- 所有窗口共享这个值
- 任一窗口"连接" → 全局变 → 所有窗口被拖去连同一个后端

后果:任意时刻所有窗口被迫连同一个后端,**无法同时开本地窗口 + 远程窗口**。

这违背原则4(软件定义书 L107):
> 应用 ≠ 窗口。**窗口是临时的观察工具**,应用是承载所有 session 的守护进程。

远程后端语境下,**daemon 才是"应用",窗口才是"观察窗"**(ADR-014 原话)。所以"每窗口连一个后端"才是原则4 的正确落地,全局 active 是偷偷把"应用"和"窗口"又绑回去了。

---

## 二、新模型

**每个窗口独立连一个后端。后端是窗口的属性,不是全局状态。**

```
[OS 窗口 1: backend=null(本地)]    → 本地 main 后端
[OS 窗口 2: backend=profileX]      → 远程 daemon X
[OS 窗口 3: backend=profileY]      → 远程 daemon Y
```

- 三个 OS 窗口同时开,各连各的 daemon,互不干扰
- 每个窗口的 session/path/settings **来自它连的那个后端**(snapshot 独立)
- 切换后端 = **开新窗口**(符合"窗口零成本开关"),不是 reload 当前窗口
- **不再有"全局 active"** —— 删掉这个概念

### 数据归属与跨后端语义
- 本地窗口的 session 属于本地 main;远程窗口的 session 属于该远程 daemon
- 本地窗口看不到远程窗口的 session(不同后端,不同 snapshot)—— 这是正确的(数据归 daemon)
- 同一 daemon 上的本地窗口与远程 WS client 共享 path/session 树,session owner 使用统一 ownerClientId
- `focus-owner` 不能假设 ownerClientId 一定是 daemon 本地 BrowserWindow id:本地 owner 由 daemon 直接聚焦;远程 owner 通过定向事件让客户端本地 Electron 聚焦

### 控制面 / 数据面边界(0.2.4-remote-test.4 勘误)

每窗口 backend 不是“这个窗口的所有 API 都发给远端”。必须拆成两条路由:

- **客户端本地控制面**:`APP_QUIT`、`WINDOW_*`、远程 profile 凭据管理、`WINDOW_MAX_STATE_CHANGED`。这些能力控制当前机器上的 Electron/BrowserWindow,永远走本地 IPC。
- **backend 数据面**:session/path/bookmark/template/settings/file-panel 等业务状态。远程窗口走 WS,本地窗口走本地 IPC。

远程窗口发普通 `WINDOW_CREATE` 时,新窗口默认继承当前 backendProfileId;否则会错误创建成本地后端窗口,表现为新窗口左侧没有远程 path/session。显式指定另一个 profile 时以显式值为准。

`SESSION_OPEN_IN_NEW_WINDOW` 是跨控制域复合操作,不能整体发给 daemon。远程模式必须按以下顺序执行:旧 remote client 在 daemon 释放 session → 客户端本地 main 创建携带同一 backendProfileId + selectSessionId 的 BrowserWindow → 新窗口连 daemon 后用新 clientId claim orphan session。daemon 绝不能替客户端创建物理窗口。

启动同步采用“先订阅事件、后拉 snapshot”。snapshot 与事件共用同一有序 transport:请求前的状态包含在 snapshot,响应后的变化由 listener 接住,避免旧的 snapshot→subscribe 空档丢失 path/session 事件。

## 六、为什么优先于阶段3

阶段3(断连重连 / session 接管)建立在"窗口↔后端"关系上。关系错了(全局 active),阶段3 的重连/接管语义会跟着歪(重连哪个后端?接管哪个 daemon 的 session?)。先把关系理对,阶段3 才有干净的地基。
