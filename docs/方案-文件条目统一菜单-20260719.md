# 方案：文件条目统一菜单抽象（ADR-018）

**日期**：2026-07-19
**版本**：v0.3.1
**状态**：✅ 已落地
**关联**：ADR-018（软件定义书）、ADR-017（v0.3.0 FileListRow 渲染统一）

---

## 背景

v0.3.0（ADR-017）把三面板文件条目的**渲染**统一成 `<FileListRow>`，但**右键菜单**仍是各面板 `items.push(...)` 自拼。共享 builder（`copyPathItems` / `revealInExplorerItem`）名义上复用，实际：

- **file-panel** 真用了共享 builder
- **file-tree** 只持 `rootId + relativePath`，拿不到绝对路径 → 不能用 `revealInExplorerItem(absolutePath)` → 自己手写一套（走专用 `reveal-path` IPC）
- **git** 只持 `relativePath`（repoRoot 在 main 端，故意不暴露给 renderer）→ 同样不能共享 → 自己手写

结果：同样的「复制路径 / reveal」在三处实现不一致；新增能力（如 git 的「打开文件本身」）要改 N 处。开发者明确：「**以后会有更多面板的**」——当前抽象不彻底。

## 决策：能力驱动的统一生成器

引入 `buildFileEntryMenu(ctx, deps)`（`src/renderer/components/common/fileListRowContextMenu.ts`）。

### FileEntryContext —— 条目的能力清单

```ts
interface FileEntryContext {
  primary?: { label; run };              // 主操作(左键/双击)
  openFile?: { run; disabled? };          // 打开文件本身(与 primary 不同时,如 git)
  relativePath?: string;                  // → 自动生成「复制相对路径」
  resolveAbsolutePath?: () => Promise<string>; // → 自动生成「复制绝对路径」
  reveal?: () => void;                    // 在 Explorer 显示(各面板自实现)
  close?: { close; closeOthers; closeAll; closeOthersDisabled?; closeAllDisabled? };
}
```

**全可选**（除语义上的 primary 推荐）。提供哪些能力 → 生成哪些菜单项。

### 统一菜单形态

```
操作族(primary / openFile / close) → divider → 路径族(copyRelative / copyAbsolute / reveal)
```

两组都非空才插 divider，尾部不插。文案统一（`deps.tx` 双语）。

### 关键：能力驱动 ≠ 项数一致

- 目录 / file-panel tab 无「打开文件」→ 不提供 `openFile` → 菜单没这项（合理，不强求）
- file-tree 保持 rootId 抽象 → 不提供 `resolveAbsolutePath` → 没「复制绝对路径」（合理）
- 各面板菜单「能做什么」由其**本性**决定，生成器只负责把这些能力以统一形态呈现

### 关键：路径解析保持各面板特色

`reveal` / `openFile` / `resolveAbsolutePath` 是**注入的能力**，不强制统一实现：

| 面板 | reveal 实现 | resolveAbsolutePath |
|---|---|---|
| file-tree | `cmd:file-tree:reveal-path`（rootId + relativePath，main 端 resolve + show） | 不提供（rootId 抽象，不暴露绝对路径） |
| git | `resolve-path` → `cmd:system:show-in-explorer` | `cmd:git:resolve-path`（repoRoot 在 main） |
| file-panel | 直接 `cmd:system:show-in-explorer`（已有绝对路径） | `async () => file.path`（同步） |

三套路径来源（git repoRoot / file-tree root / file-panel 绝对）本质不同，**硬统一会引入不必要的抽象层**。统一接口 + 注入实现，是正确层次。

## 为什么不把路径也统一成绝对路径

考虑过让所有面板 resolve 出绝对路径再走单一 `SYSTEM_SHOW_IN_EXPLORER`，但：

1. file-tree 的 rootId 抽象是刻意的（root 可为 workspace 虚拟根等），向 renderer 暴露绝对路径破坏抽象
2. 三套路径来源本质不同，统一层只增加间接调用而无真实简化

能力接口统一 + 实现注入，既统一了形态（用户可见的一致性），又尊重了各面板的路径模型。

## 接入新面板（未来面板指南）

```tsx
import { buildFileEntryMenu, type FileEntryContext } from '../common/fileListRowContextMenu';

function MyPanel() {
  const copyToClipboard = useCopyToClipboard();
  const toast = useToast();
  const { tx } = useTranslation();

  const ctx: FileEntryContext = {
    primary: { label: tx('打开', 'Open'), run: () => openMyFile(entry) },
    relativePath: entry.relativePath,
    reveal: () => myRevealLogic(entry),
    // openFile / resolveAbsolutePath / close 按需提供
  };

  return (
    <FileListRow
      variant="list"
      label={entry.name}
      onClick={...}
      buildContextMenu={() =>
        buildFileEntryMenu(ctx, {
          copyToClipboard,
          toastError: (m) => toast.push({ kind: 'error', message: m }),
          tx,
        })
      }
    />
  );
}
```

**零拼装菜单**：填能力 → 用 FileListRow → 完成。

## 影响

- file-tree / git / file-panel 三处 `buildContextMenu` 改用 `buildFileEntryMenu`
- 删除冗余的 `copyPathItems` / `revealInExplorerItem`（被生成器取代）
- 菜单文案统一（此前 file-panel 叫「复制路径」，其他叫「复制相对路径」，现按能力统一）
- 未来面板按上文接入指南实现，零拼装
