## I. 定位与核心不变式

### I.1 总体定位

**Marina = 本地终端管理器,可选启用"远程后端模式"把 daemon 部署到另一台机器,本地 UI 作为 client 连过去。**

远程后端模式下:
- **后端 daemon**(Main Process:PTY / Session / Path / FilePanelService / SSH / 持久化)跑在**远程机器 B**
- **UI client**(Renderer)跑在**本地笔记本 A**
- 两者通过 **WebSocket + TLS + token** 通信,走 WireGuard 虚拟局域网

用户的核心诉求(已确认):
- **b**:要"远程机器上的完整 Marina 体验"——不止远程 shell,还要 Path 收藏 / 模板 / 设置 / file-panel 一整套
- **d**:远程是 Windows,SSH 体验差(无 tmux、ConPTY-over-SSH 别扭、sshd 配置麻烦)
- **file-panel 远程可用**:远程 agent 把它正在处理的文件打开到本地用户眼前的侧栏

### I.2 为什么不是 SSH 方案(§14)

SSH 方案(§14,M1 已落地)的本质是**本地 Marina spawn ssh 进程连远程 sshd**,它的能力边界是"远程 shell"。满足不了本场景的三条理由:

1. **b 不满足**:SSH 给的是远程 shell,Path 收藏 / 模板 / 设置这些数据**全在本地 Marina**,远程机器上没有任何 Marina 状态。所谓"完整体验"在 SSH 模型里不存在。
2. **file-panel 架构性失效**:`src/main/file-panel-service.ts` 的 `@SSH 限制` 注释已说死——远程 agent 拿到的 `MARINA_SERVICE=http://127.0.0.1:PORT` 指向本地,远程机器到不了;且 `fs.readFile`/`fs.watch` 跑在本地,远程路径在本地不存在。**功能完全缺失,不是体验差**。要让它在 SSH 下工作,必须引入 SFTP → 直接碰 §14.6"不做文件管理"红线,且 `fs.watch` 远程做不到,自动刷新残缺。
3. **d 共振**:远程 Windows + SSH 本来就是已知痛点,强化不走 SSH。

**关键洞察**:file-panel 的本机机制(HTTP + 本机 fs)**天然要求"daemon 和 agent 同机"**,这恰好就是远程后端方案的定义。file-panel 不是"额外诉求",反而是**支持远程后端方案的最强论据**。

### I.3 与 SSH 方案的并存关系

SSH 方案**不废弃**,定位调整为"远程后端不适用时的轻量回退":
- 临时连一台**没装 Marina** 的机器(别人的服务器、跳板机)
- 只需要 shell、不需要 file-panel / 完整体验

两者共用 Path 的 `kind` discriminated union:
- `kind: 'local'` — 本地后端
- `kind: 'ssh'` — SSH 终端(§14,远程 shell)
- `kind: 'remote-daemon'`(新增)— 远程后端模式(远程 Marina,完整体验)

### I.4 本地用户感受不变式

**不开远程后端的用户体验与今天 100% 一致。**

- 不创建 remote-daemon 连接、不在"高级 → 启用远程后端"勾选 → UI 看不到任何远程后端元素
- daemon 化的传输层改动是**接缝级**的,本地 in-process client 走原 Electron IPC 路径,零开销
- 启动时间 / 内存基线不变(WS 传输层 lazy load,无远程连接时不消耗)
- 由 invariant 测试守护(对齐 §14.2 的本地视野守护思路)

## II. 架构

### II.1 现状:单进程,前后端已分离

Marina **现在已经是前后端分离结构**,只是前后端跑在同一个 Electron 进程:

```
┌─────────────────── 一个 Electron 进程(一台本地机器)──────────────────┐
│                                                                       │
│  Renderer(前端层,每窗口一个)                                          │
│   React UI: Sidebar / Tabs / TerminalView(xterm)/ Settings / FilePanel│
│   store + 视图逻辑 + xterm 装配                                         │
│            │                                                           │
│       window.api  ◄── preload「最薄的桥」(src/preload/index.ts)        │
│            │  invoke / on(只暴露语义,不碰 ipcRenderer)                 │
│  ──────────╪────────────────── Electron IPC(同进程)─────────────────  │
│            │                                                           │
│       ipc.ts  ◄── ipcMain.handle + wireEventBroadcasts(src/main/ipc.ts)│
│  Main(后端层):SessionManager / PathManager / FilePanelService /        │
│               SSH 全套 / SettingsManager / Persistence / PlatformAdapter│
└───────────────────────────────────────────────────────────────────────┘
```

**三个证据证明接缝已经预留好**:
1. `src/preload/index.ts` 文件头原话:"**这里不写业务逻辑,只是一座最薄的桥**"。renderer 只依赖 `window.api.invoke/on` 的**语义**。
2. `src/main/ipc.ts` 集中注册所有 `ipcMain.handle` + 一个 `wireEventBroadcasts` 统一路由事件。
3. `docs/ipc-protocol.md`(1860 行)第 2 章标题就叫 **「传输层」**——协议设计者当初就有意识地把传输层和语义层分开。

### II.2 目标:双进程跨网络,前后端层整体复用

```
  笔记本 A(Electron client)                  远程 Windows B(Marina daemon)
 ┌─────────────────────────────┐             ┌─────────────────────────────┐
 │ Renderer(前端层)            │             │ Main(后端层)                │
 │  🟢 整体复用,不改            │             │  🟢 整体复用,不改            │
 │      │                       │             │      ▲                       │
 │  window.api                 │             │   ipc.ts                    │
 │      │                       │             │      │                       │
 │  ┌───┴──────────────┐        │  WS + TLS   │  ┌───┴──────────────┐        │
 │  │ invoke/on 实现   │         │ ──────────► │  │ handle/send 实现 │        │
 │  │ 🔧改成 WS client │ ════════╪═════════════╪═►│ 🔧改成 WS server │        │
 │  └──────────────────┘        │  + token    │  └──────────────────┘        │
 └─────────────────────────────┘             └─────────────────────────────┘
        🔧 改接缝                                      🔧 改接缝
   ➕ Electron client 入口                     ➕ daemon 入口 / 认证 / 重连
                                              ➕ 同时服务 in-process UI(双模式)
```

**核心**:Renderer 整体复用(不只是"渲染层",是整个前端层:React 组件 + store + 视图逻辑 + xterm 装配 + FilePanel + MarkdownViewer)。Main 后端逻辑整体复用(SessionManager / PathManager / FilePanelService / SSH / Settings / Persistence / PlatformAdapter 全部不动)。**真正动的只有中间的传输层接缝 + 新增 daemon/client 两个壳。**

## IV. IPC 协议演进

### IV.1 传输层升级(Electron IPC → WS + TLS)

现有 `ipc-protocol.md` §2「传输层」只描述了 Electron IPC。升级后支持两种传输:
- **Transport-Local**:Electron `ipcRenderer.invoke` / `ipcMain.handle`(本地 in-process client 用,零改动)
- **Transport-Ws**:WebSocket + TLS(远程 client 用,新增)

两种 transport 上跑**同一套语义**(命令 / 事件 / 信封 / 错误码 / 时序约束)。`ipc-protocol.md` 的 §3-§9(语义层)几乎不动,只重写 §2(传输层)加 Transport-Ws 子章。

### IV.2 信封升级(`windowId` → `clientId`)

现有信封(§2.3 / §2.4):
```
命令信封: { windowId, requestId, payload }
事件信封: { eventId, timestamp, payload }
```

问题:本地模型里"窗口 = IPC 连接的端点";远程模型里一个 client 可能对应 daemon 侧的多个窗口视角,且 client 会断线重连(windowId 会变,session 不应变)。

升级:
```
命令信封: { clientId, requestId, payload }   // clientId 稳定标识一个 client 会话
事件信封: { eventId, timestamp, payload, targetClientId? }
```

`clientId` 在握手时分配,断线重连时**复用同一 clientId**(凭 token 认主),session 的 owner 用 clientId 而非 windowId 记录。

### IV.3 新增语义:claim / release / reconnect

这是新协议相比现有 IPC **唯一真正新增的语义**(ADR-005 单 owner 约束在网络下的延伸):

- `cmd:session:claim` — client 请求成为某 session 的 owner(已被占则拒)
- `cmd:session:release` — client 主动释放 ownership(切到别的 session / 关闭)
- **自动 release**:client 断线 → daemon 检测 WS 关闭 → 该 client 持有的所有 session 自动 `ownerClientId = null`(session 不销毁,scrollback 保留)
- 重连:同一 clientId(token 认主)重连后,可重新 claim 原 session

### IV.4 双模式 dispatcher

daemon(B 上的完整 Marina)同时服务两类 client:
- **in-process Renderer**(B 本地用户,B 上开的窗口)→ 走 Transport-Local
- **远程 WS client**(笔记本 A)→ 走 Transport-Ws

实现:把 ipc.ts 的 handle 抽成纯函数 `handleCommand(source, envelope) => result`,前面挂一个 dispatcher,两种 transport 都汇到它。事件广播也统一:`broadcastEvent(targetClientIds, envelope)`,dispatcher 决定每个 target 走 IPC 还是 WS。

## V. 认证与安全

### V.1 分工:WireGuard 管可达性,Marina 管认证 + 加密

| 层 | 负责方 | 内容 |
|---|---|---|
| 可达性(两机能互通) | **WireGuard**(用户已自建) | 虚拟局域网,远程机器拿到稳定内网 IP |
| 加密(传输防窃听) | **Marina TLS** | daemon 监听 `wss://`,自签证书或 WireGuard 内 CA 签 |
| 认证(谁允许连) | **Marina token** | daemon 持久化生成,client profile 里存;首次配对走本地 UI 展示 token + 二维码 |

**为什么 TLS 不能省**(已经有 WireGuard 加密了):纵深防御。WireGuard 是用户自管的网络层,Marina 不应假设它一定在;且 TLS 提供 daemon 身份验证(防连错机器),WireGuard 不提供应用层身份。

> **v2.0 阶段2 实现决议(ADR-015)**:阶段2 用纯 token + `ws://`,**暂不实现 TLS**。理由:(1) V1 目标场景为 WireGuard 内网(本方案 §VI 决策),WG 已提供传输加密;(2) TLS 自签证书生成需新依赖(node 无原生 X509 证书生成 API,forge/selfsigned 违反 AGENTS.md 边界2),手搓 ASN.1 成本高;(3) token 认证已防网内误连。TLS 延后至跨网络/公网场景(V2),届时连同指纹确认(§14.3 known_hosts 风格)一并做。本节"TLS 不能省"的论证在 V2 跨网络场景下仍然成立。

### V.2 威胁模型

- ✅ 防:网络窃听(TLS)、未授权连接(token)、连错 daemon(TLS 证书指纹 + token 双校验)
- ✅ 防:远程 agent 越权读文件(file-panel 已有的"仅限已打开列表路径"+ session 绑定机制不变)
- ⚠️ 不防:拿到 token 的恶意 client 冒充(等同拿到 SSH key)——靠 token 不泄露 + 可在 UI 一键吊销
- ⚠️ 不防:daemon 机器被 root——超出威胁模型,等同任何本地工具

### V.3 配对流程(首次把笔记本 A 接到 daemon B)

1. B 上打开 Marina(本地 UI),设置 → 远程后端 → "启用 daemon",生成 **token + 二维码**(二维码含 `wss://<B的WG-IP>:<port>?token=...`)
2. A 上打开 Marina → 新建 remote-daemon profile → 扫码 / 粘贴
3. A 的 client 握手 → TLS 指纹展示给用户确认(对齐 §14 的 known_hosts 体验)→ 通过 → daemon 分配 clientId 一并返回 → 后续自动连

### V.4 token 与 clientId 的生命周期(决议问题 2)

**决策:`safeStorage` 持久化,daemon 端 + client 端各一份。**

- **token 在 daemon 端持久化**:存 `marina-daemon-credentials.json`,用 Electron `safeStorage` 加密(Windows 走 DPAPI,对齐 §14.7 SSH 密码方案)。**daemon 重启 token 不变 → client 重连无需重新配对**。
- **可吊销**:daemon 端 UI(设置 → 远程后端)提供"重置 token" → 旧 token 立即失效,所有 client 需重新配对。
- **clientId**:首次配对握手时由 daemon 分配(UUID),随 token 一起返回,client 持久化。断线重连凭 token 认主,复用同一 clientId。
- **client 端存储**:remote-daemon profile `{ daemonUrl, displayName, token, clientId }`,整体 safeStorage 加密。
- **选 safeStorage 而非 keytar**:与 §14.7 SSH 密码存储一致,**无新原生依赖**,符合"当前 Marina 很轻量"的用户诉求。

## VI. 双模式 daemon 设计

### VI.1 一份 Marina.exe,两种身份 + 登录自启(决议问题 1)

用户已定:**远程跑完整 Marina**(因为"当前 Marina 很轻量")。所以 `Marina.exe` 同时是:
- **本地应用**(B 上有人时):开 BrowserWindow,本地用户正常用
- **daemon**(B 上没人时,或同时):监听 WS,服务远程 client

**启动模式(决议:登录时自启 detached,非 Windows Service)**:
- daemon **不做** Windows Service(避免管理员权限安装 + Service 跑 SYSTEM 账户拿不到用户 shell / 凭据)。
- 改为**登录时自启 detached 进程**:安装时(或设置页开关)用 Windows 任务计划程序(`schtasks`)注册"登录时"触发,命令行 `Marina.exe --headless --daemon`。
- `--headless`:不开 BrowserWindow;`--daemon`:开 WS 监听。
- **登录态活**:符合 spec L2012 daemon 候选"detached process,登录态活"设计;PTY 继承用户环境变量(home / PATH / 凭据),这是 file-panel / SSH 能正确工作的前提。
- 仍是**一个 Marina.exe 产物**,通过启动参数决定身份。保留 ADR-002"一个产物"精神。

> **术语澄清**:用户口语说"开机自启",技术上 Marina daemon 跑**用户登录态**(非 SYSTEM 账户),所以是"登录时自启"。Windows Service 跑 SYSTEM 账户反而拿不到用户 shell / 凭据,不适合本场景。

### VI.2 in-process client + 远程 WS client 共存

B 本地用户和 A 远程用户**可以同时用同一个 daemon**。此时:
- B 的本地 session:B 的 in-process Renderer 是 owner
- A 的远程 session:A 的 WS client 是 owner
- 互不干扰,各自看各自的 Path / session 视图
- 共享的是**同一份后端数据**(Path 收藏、SSH profile、settings)— 这正是 b"完整体验"想要的:在 B 前面加的收藏,A 连上来也能看到

### VI.3 session owner 仲裁(与 ADR-005)

ADR-005"一个 session 同时只有一个 owner"在网络下保留,owner 标识从 `windowId` 改为 `clientId`。两个 client 想同时看一个 session?按 ADR-005 现有答案:**不支持多 owner**,第二个 client 看到的是"被占用,点这里抢"(抢 = 让原 owner release)。

### VI.4 远程 client 数量上限:1(决议问题 3)

**决策:V1 daemon 同一时刻最多接受 1 个已认证远程 client。**

- 第二个尝试握手 → 拒绝并提示"已有 client 连接,如需连接请先在 daemon UI 断开现有 client"。
- daemon UI 显示当前连接的 client(clientId / 连接时间 / 远程 IP)。
- 多 client 留 V2(涉及多 owner 协调,见 ADR-005)。

### VI.5 数据归属:归 daemon 侧 B(决议问题 4)

**决策:所有数据归 daemon(B),client(A)是纯观察窗 + 输入通道。**

- A 通过 B 创建的 Path 收藏 / SSH profile / settings / templates → 存 **B** 的 `%APPDATA%/Marina/`
- A 自己也是完整 Marina:A 不连 B 时,用自己的本地数据;A 连 B 时,**整个后端数据源切换到 B**(不是合并,是切换)。
- **语义类比 SSH client 连不同 server**:连哪台看哪台的数据。
- **含义**:A 离线时看不到"通过 B 创建的历史"——可接受(B 才是主数据源;A 是接入点)。
- **收益**:B 前面加的收藏,A 连上来立刻看到;符合 b"完整体验"诉求。

## VII. ADR-014 草案

> **ADR-014:为什么从"单进程 Electron"演进到"daemon + 网络 client"**
>
> **状态**:草案,待立项
>
> **背景**:ADR-002 当年选择 Electron 原生 IPC 而非独立守护进程,理由是 (1) Electron Main 本身就是守护进程,(2) Renderer 与 Main 间 Electron IPC 极快,(3) 简化部署:一个 .exe 即所有,(4) 关窗存活靠托盘解决。
>
> **变更**:新增"远程后端模式",Main 可作为 daemon 监听 WS,接受远程 Electron client 连接。
>
> **为什么反转 ADR-002 的部分前提**:
> - (1) 仍成立:daemon 还是 Electron Main,不是独立技术栈
> - (2) 仍成立:本地 in-process client 继续走 Electron IPC,零开销;只有远程 client 走 WS
> - (3) 仍成立:仍是**一个 Marina.exe 产物**,通过启动参数(`--headless --daemon`)决定是否开 daemon 监听
> - (4) 扩展:关窗存活 / 崩溃存活(原 daemon 候选的诉求)+ **跨机器访问**都由这个架构覆盖
>
> **触发理由**:用户需求 b(远程完整体验)+ d(Windows 远程)+ file-panel 远程可用,三点统一指向"daemon 和 agent 必须同机",SSH 方案做不到(见 §I.2)。这提前启动了 spec L2012 的 daemon 候选,并把它从"本地 named pipe"扩展到"跨网络 WS"。
>
> **代价**:IPC 传输层重写(接缝级,语义层不动)+ 自做认证加密(sshd 白送的能力)+ 断连重连 / session 接管新语义。业务逻辑零改动。
>
> **不做的**:
> - 不做 NAT 穿透 / 中心 relay(用户用 WireGuard 自组网)
> - 不做 Web UI(V2 候选,V1 只 Electron client)
> - 不做多 owner(保留 ADR-005)
> - 不做 Windows Service(用登录自启 detached,见 §VI.1)

## XI. 决议记录(2026-07-05 全部拍板)

| # | 问题 | 决议 | 理由 |
|---|------|------|------|
| 1 | daemon 启动模式 | **登录时自启 detached**(`Marina.exe --headless --daemon`,任务计划程序注册),非 Windows Service | PTY/shell 跑用户登录态才拿得到 home/PATH/凭据;Service 跑 SYSTEM 反而拿不到。对齐 spec L2012"detached,登录态活"。避免管理员权限安装 |
| 2 | clientId + token 存储 | **safeStorage 加密持久化**(daemon 端 + client 端各一份) | 与 §14.7 SSH 密码方案一致,**无新原生依赖**;daemon 重启 token 不变 → client 无需重新配对;可一键吊销 |
| 3 | 远程 client 数量上限 | **1 个** | 够用且简单;多 client 涉及多 owner 协调(ADR-005),留 V2 |
| 4 | 数据归属 | **归 daemon(B)**,client 是观察窗 | 符合 b"完整体验"(B 前加收藏 A 立刻见);A 离线看不到历史可接受 |
| 5 | file-panel 大文件 over WS | **复用现有上限**(text 2MB 截断 / image 10MB 拒绝),base64 over WS,V1 不优化 | 上限已合理;WireGuard 下 13MB base64 无压力;自动刷新链路 over WS 天然工作(daemon fs.watch → 事件 → client mtimeMs 变 → 重读) |
