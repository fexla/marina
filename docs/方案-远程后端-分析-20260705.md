# Marina 远程使用方案 · 阶段分析

> **状态**:讨论草稿(2026-07-05)。本文由 agent 沉淀,用于在 file-panel 内阅读。
> 决定方向后再正式立项 → `docs/方案-远程后端-YYYYMMDD.md` + 新 ADR。

---

## 0. 需求原话

> 做一个远程使用 Marina 的功能,通过 Marina 和另一台电脑上的 CLI 程序交互。

用户在两轮澄清后明确了三点诉求:

- **b. 想要"远程机器上的 Marina 完整体验"** — 不止远程 shell,还要 Path 收藏 / 启动模板 / 设置 / file-panel 那一整套。
- **d. 远程是 Windows,SSH 体验差** — OpenSSH server on Windows 配置麻烦、没有 tmux、ConPTY-over-SSH 体验差。
- **file-panel 要在远程可用** — 远程机器上跑的 CLI agent 能把它正在处理的文件,打开到本地用户眼前的侧边面板,体验和本地一致。

---

## 1. 现状:SSH 方案(§14)的边界

项目已有一份定型的远程方案 —— `docs/方案-SSH-完整支持-20260524.md`(v2.1),已写进 spec §14 并落地到 v0.2.x(M1 = 0.2.0 GA 候选)。它的本质是:

> **本地 Marina**(UI + 后端都在本地)spawn 一个 `ssh` 进程,通过 sshd 连到远程机器的 shell,在远程 shell 里跑任意 CLI 程序。

已经做到的:profile 管理、ssh_config 集成、ControlMaster 连接复用、ProxyJump 跳板、known_hosts、远程 tmux attach-or-create、重连按钮。

但它满足不了用户的 b + d + file-panel:

| 诉求 | SSH 方案给不给得了 |
|---|---|
| **b** 远程机器上的"完整 Marina 体验" | ❌ SSH 给的是"远程 shell"。Path 收藏 / 模板 / 设置这些数据**全在本地** Marina,远程那台机器上没有任何 Marina 状态。所谓"完整体验"在 SSH 模型里根本不存在 |
| **d** 远程 Windows + SSH 体验差 | ✅ 真实痛点,且强化了"不该走 SSH"的判断 |
| file-panel 远程可用 | ❌ 见下节,架构性不工作 |

## 2. file-panel 在 SSH 下为什么不工作

`src/main/file-panel-service.ts` 的文件头注释里,作者已经把这件事说死了(`@SSH 限制` 段):

> SSH 会话的 currentCwd 是远程路径,且**远程进程根本到不了本机 127.0.0.1**(除非反向隧道,超出 v1)。所以本功能 v1 **仅实质支持本地终端**;即便 SSH 程序误调,fs.stat 远程路径会失败 → 返回错误,安全无副作用。

不是"体验差",是**功能完全缺失**。两个架构性原因:

1. **控制通道断了**:远程 agent 拿到的 `MARINA_SERVICE=http://127.0.0.1:PORT` 是**本地** Marina 的地址,远程机器访问不到。
2. **数据通道断了**:就算控制通道能通,`fs.readFile` / `fs.watch` 跑在**本地** Marina,操作本地文件系统,**远程路径在本地根本不存在**。

> 注:file-panel 本身**不违反 §14.6**。§14.6 禁的是"文件管理 / SFTP 浏览 / 远程编辑 / 拖拽上传",而 file-panel 是 "agent 主动把它正在处理的文件推给用户看",属于终端输出的延伸,只读、不写、不传。

## 3. file-panel 远程化的三条技术路径

让远程 agent 把文件打开到本地侧栏,理论上有三条路:

### 路径 A — SSH 反向隧道(`-R`)

把本地 file-panel 的 HTTP 端口反向映射到远程,让远程 agent 能调到本地 Marina。

- 控制通道能通,但 `cmd:file-panel:read` 仍在本地 `fs.readFile` 远程路径 → 失败。
- 要修就得**额外引入"读远程文件内容"的能力 = SFTP** → **直接碰 §14.6 红线**。
- 且 `fs.watch` 远程文件做不到(SFTP 没有 inotify),**自动刷新功能残缺**。

### 路径 B — 远程跑半个 file-panel agent

在远程机器上跑一个小进程,监听远程 agent 的 HTTP、本地 fs 操作远程文件、再通过 SSH 通道把内容推回本地 Marina。

- 等于在远程**部署了 Marina 后端的一部分**(file-panel-service 的远程副本)。
- 既然都要在远程跑后端了,不如直接跑完整后端 → 退化成路径 C。

### 路径 C — 远程后端方案(daemon 网络化) ⭐

Marina 的 Main Process 整个跑在远程电脑 B,本地只是 UI client 连过去。

- **file-panel-service 在远程 daemon 里**,`fs.readFile` / `fs.watch` 都是远程的本地操作,**零障碍**。
- `MARINA_SERVICE=127.0.0.1:PORT` 注入给远程 agent,远程 agent 和远程 daemon 同机,**loopback 天然可达**。
- 文件内容 / 状态变化由远程 daemon 经网络推给本地 UI 渲染。
- **不碰 §14.6 红线**(没有引入 SFTP,远程 fs 操作是远程 daemon 自己的本地操作)。

### 关键洞察

> file-panel 的远程体验,在 SSH 方案下**要么破红线、要么退化成"半个远程后端"**;而在远程后端方案下**几乎是免费的** —— file-panel-service 代码几乎不用改,只是输出从"本地 IPC"换成"网络推给 client"。
>
> 也就是说,file-panel 不只是"额外诉求",它反而成了**支持远程后端方案的最强论据**:它的架构(本机 HTTP + 本机 fs)天然要求"后端和 agent 在同一台机器",这恰好就是远程后端方案的定义。

## 4. 结论:b + d + file-panel 三点统一指向"远程后端"

三个诉求互相加强,结论清晰:**应该做远程后端(Marina daemon 网络化 + UI client),不该走 SSH。**

这等于提前启动 spec 末尾的 **daemon 候选(v2.0+)**,并把它从"本地 named pipe"升级到"跨网络"。代价:

- 推翻 **ADR-002**("为什么 IPC 用 Electron 原生而不是独立守护进程")。
- 重写 IPC:从 Electron IPC → 网络协议(WebSocket / TLS),加断连重连、session 接管。
- 自己做安全:传输加密、认证、多 client 隔离(SSH 方案这些是 sshd 白送的)。
- 部署:从"一个 .exe"→ daemon + UI 双产物。

量级需要一个正式方案文档 + 新 ADR 来定,不能直接动手。

## 5. 待拍板问题(回答后我写正式方案)

1. **UI client 形态**:先只做 Electron client(复用 ~90% renderer),还是 Electron + Web 都要?
   - 建议:**先 Electron client,Web 留 V2**。两者共用同一套后端网络协议,先做 Electron 不会白做。

2. **远程 daemon 是"完整 Marina"还是"精简 headless 后端"?**
   - 完整:远程 Windows 上也装一份 `Marina.exe`,既能本地用、也能被远程 client 连(双模式)。
   - 精简:单独一个 `marina-daemon.exe`,无 UI,只服务远程 client。
   - 建议:**完整双模式**。用户在远程机器前时用本地 UI,离开时用笔记本 / iPad 连同一个 daemon,session / Path / file-panel 全部连续。最符合"窗口平等、应用 = 守护进程"的哲学。

3. **连接方式**:局域网直连(TLS + token)够不够?要不要穿透 NAT(类似 VS Code Tunnels / frp)?
   - 建议:**V1 只做局域网 / Tailscale 这类已组网场景**。NAT 穿透工作量大,留 V2。

## 6. 对现有 spec / ADR 的影响(预览)

正式立项时需要同步修订:

- **新增 ADR-014**:为什么从"单进程 Electron"演进到"daemon + 网络 client",反转 ADR-002 的哪些前提。
- **§9 技术架构**:补充 daemon + client 的双产物架构图。
- **§9.4 IPC 协议**:升级为网络协议(WebSocket / 自定义 TLS),支持断连重连。
- **§13.2 永远不做**:确认"远程后端"不在禁止列表(目前没有相关条目)。
- **§14 SSH 模式**:定位调整为"远程后端不适用时的轻量回退"(例如临时连一台没装 Marina 的机器)。
- **daemon 候选(L2012)**:从"v2.0+ 评估"提前到"当前立项",并扩展到跨网络。
