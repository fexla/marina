# 方案:面板系统改进(icon / 顶部切换 / UI 状态持久化 / 缩进统一)

> 日期:2026-07-21 · 状态:**待开发者评审** · 涉及版本:`0.3.1-dev.1` → `[Unreleased]` 积累
>
> 本文档是开发计划,不是已实现的设计。评审通过后再进入实现。
> 实现时同步落一份 ADR(建议顺延为 ADR-019,实现时确认编号)固化"面板 UI 状态分层"与"缩进单一真相源"两条决策。

## 0. 背景

开发者提了 4 个面板改进需求:

1. 文件条目 icon 根据后缀名变化
2. 文件面板的「当前目录」和「临时工作区」改为类似 Git 面板顶部按钮切换(现在两个并排)
3. 文件面板目录展开状态切面板再切回会丢;**其他面板也有类似问题,要系统性解决**
4. 文件面板和 Git 面板层级缩进不一致(文件面板的更好);**要系统性解决,避免新面板重蹈**

需求 3、4 明确要求**系统性方案**。本方案的核心价值不在"修好 4 个点",而在**建立共享基础设施**,让这 4 个需求成为基础设施的首批消费者,未来新面板零成本复用、不会再出同样问题。

## 1. 现状诊断

### 1.1 各需求的直接根因

| 需求 | 直接根因 | 关键代码 |
|---|---|---|
| 1. icon 不分后缀 | 所有文件条目硬编码 `icon="file"`;无扩展名→icon 映射 | `FileListRow` 调用方(FileTreePanel/GitPanel/GitTree/FilePanel)全传 `'file'` |
| 2. 目录/工作区并排 | `FileTreePanel` 用 `roots.map(root => <section>)` 渲染双 root section | `FileTreePanel.tsx` 根渲染段(约 310-341 行) |
| 3. 展开态切面板即丢 | 展开态存组件内 `useState`,`<ActivePanel key>` 卸载即丢 | `FileTreePanel` 的 `directories` state、`GitTree` 的 `collapsed` Set |
| 4. 缩进不一致 | FileTree 走 **CSS 层叠**(`.file-tree-entry .file-tree-entry { margin-left:14px }`),Git/FilePanel 走 **inline style**(`FileListRow` 的 `depth*14`);14px 是约定但无单一真相源 | `global.css` 行 1838-1839;`FileListRow.tsx` 行 154 |

### 1.2 共同病根(系统性视角)

面板系统目前**缺三样东西**:

1. **统一的"UI 状态层"**。各面板 UI 状态四散:
   - 瞬态(loading/error)→ `useState`(丢了合理)
   - 工作态(展开目录、选中项)→ `useState`(**丢了不合理** ← 需求 3 痛点)
   - 偏好(视图模式)→ 散落的 `localStorage`(`marina.git.viewMode` / `marina.sidebar.segment` / `marina.sidebar.width`,各自读写、key 不统一)
   
   已有先例 `git-status-cache.ts`(模块级 Map,跨 mount 保留 snapshot),但**只 Git 面板用、只缓存数据**。没有通用框架。

2. **缩进/icon 的"单一真相源"缺失**。14px 缩进在 CSS 层叠、inline style、`.git-panel-group-entries` 三处硬编码;文件 icon 全是 `'file'` 一个名,扩展名分类逻辑 `file-kind.ts` 已经存在却没接到 icon。

3. **新面板没有可循的规范**。"我的展开态该存哪、缩进该怎么写、icon 怎么传"没有决策依据,只能照抄某个老面板——抄哪个就继承哪个的毛病。

> **结论**:4 个需求是同一个病根(面板系统缺共享基础设施)的不同症状。系统性地治,4 个需求一起好,且新面板不再犯。

## 2. 系统性方案设计(核心)

### 2.1 面板 UI 状态三层模型 + 两层基础设施

把面板 UI 状态显式分三层,每层有明确归属:

| 层 | 例子 | 归属 | 生命周期 |
|---|---|---|---|
| **L0 瞬态** | loading、error、滚动中高亮 | `useState` | 随 mount 生灭(丢了合理) |
| **L1 工作态** | 展开的目录、选中项、过滤草稿 | **`PanelUiCache`(新建)** | 跨 mount 保留(session 生命周期内);重启可丢 |
| **L2 偏好** | 视图模式 tree/flat、活跃 root、排序方式 | **`PanelPreferences`(新建)** | 跨重启(localStorage) |

#### 基础设施 A:`PanelUiCache`(L1,组件外缓存)

**位置**:`src/shared/panel-ui-cache.ts`(放 shared 便于单测,与 `git-status-cache.ts` 同级同模式)

**API 示意**:
```ts
// 纯逻辑:模块级 Map<sessionId, Map<panelId, unknown>>
getPanelUiState<T>(sessionId, panelId): T | undefined
setPanelUiState<T>(sessionId, panelId, state): void
clearPanelUiState(sessionId, panelId?): void   // session 销毁时调,防泄漏

// renderer hook(封装 mount 读 / 每次写同步落 cache)
usePanelUiState<T>(sessionId, panelId, initial: T | () => T): [T, (next: T) => void]
```

**关键设计**:
- `usePanelUiState` 在 `useState` 初始化时先读 cache(命中则用旧值,未命中用 `initial`)——这就是"切回来恢复状态"。
- 每次 `setState` 同步写 cache(无需 unmount effect,避免卸载竞态)。
- **不改 `LayoutHost`**,保持 `<ActivePanel key>` lazy mount 不变(不碰面板切换性能机制,降低风险)。
- `git-status-cache.ts` 保持独立:它缓存的是**数据 snapshot**(swr 模式,有 main 端事件失效),与本缓存的**纯 UI 态**语义不同,合并反而混淆。两者模式一致、各司其职。

#### 基础设施 B:`PanelPreferences`(L2,跨重启偏好)

**位置**:`src/shared/panel-preferences.ts`

**API 示意**:
```ts
// key 规范统一:marina.panel.<panelId>.<key>
readPanelPreference<T>(panelId, key, fallback): T
writePanelPreference<T>(panelId, key, value): void
usePanelPreference<T>(panelId, key, fallback): [T, (v: T) => void]
```

**收编现状**:把散落的 localStorage key 统一管:
- `marina.git.viewMode` → `panelId='git', key='viewMode'`
- `marina.sidebar.segment` / `marina.sidebar.width` → `panelId='sidebar', key=...`
- 需求 2 的"活跃 root"→ `panelId='file-tree', key='activeRootId'`

> 老 key 做一次性迁移读取(读到旧 key 就迁到新规范),不破坏现有用户偏好。

#### 规范(写入 ADR + AGENTS.md)

新面板开发按下表"无脑"选层:

```
这个状态丢了用户会烦吗?
├─ 不会(loading/错误)           → useState          (L0)
├─ 会,但重启丢可接受(展开目录)   → usePanelUiState   (L1)
└─ 会,且重启也该记住(视图模式)    → usePanelPreference (L2)
```

### 2.2 缩进单一真相源

三步让缩进从此只有"一个地方改":

1. **CSS 变量**:`global.css` `:root` 加 `--tree-indent-unit: 14px`。
2. **唯一渲染入口**:`FileListRow` 的 `depth` prop 是唯一的缩进渲染点,内部 `marginLeft: calc(var(--tree-indent-unit) * depth)`。任何树形面板要缩进,**只能**用 `FileListRow` + `depth`,不许自己写 margin。
3. **移除 FileTreePanel 的 CSS 层叠**:删 `.file-tree-entry .file-tree-entry { margin-left:14px }`,`DirectoryChildren`/`FileTreeEntryRow` 递归时传 `depth`(根层 0,每层 +1)。这样 FileTree 与 Git 走同一条 depth 路径,视觉一致(需求 4 解决)。

### 2.3 文件 icon 单一数据源

- **扩展名分类已存在**:`file-kind.ts` 已有 `MARKDOWN_EXT`/`IMAGE_EXT`/`TEXT_EXT`/`DIFF_EXT`/`TEXT_NO_EXT`。**复用**,不重复维护第二份扩展名表。
- **新建映射**:`src/shared/file-icon.ts`,`fileIconFor(fileName): IconName`,把 file-kind 的分类结果 + 额外细分(代码/JSON/锁文件等)映射到 lucide icon 名。
- **icons.tsx 注册细分图标**:import `FileText`/`FileCode`/`FileImage`/`FileJson`/`FileArchive`/`FileLock` 等,挂到 `Icons` 表。
- **调用方一处改**:三面板把文件条目的 `icon='file'` 改为 `icon={fileIconFor(entry.name)}`(目录仍 `'folder'`)。

## 3. 各需求详细方案

### 3.1 需求 1:文件 icon 按后缀名

**改动文件**:
| 文件 | 改动 |
|---|---|
| `src/shared/file-icon.ts`(新) | 扩展名→IconName 映射,复用 file-kind.ts 的扩展名集合 |
| `src/renderer/components/icons.tsx` | 注册细分图标(FileText/FileCode/FileImage/FileJson/FileArchive/FileLock 等) |
| `src/renderer/components/file-tree/FileTreePanel.tsx` | 文件条目 `icon={fileIconFor(entry.name)}`(2 处:正常树 + 搜索结果) |
| `src/renderer/components/git/GitPanel.tsx`、`GitTree.tsx` | 同上 |
| `src/renderer/components/file-panel/FilePanel.tsx` | tab 的 icon 同样细分 |
| `src/shared/file-icon.test.ts`(新) | 映射单测 |

**映射覆盖**(按开发者反馈的分类,首版):

| 类别 | 代表后缀 | 拟用图标(lucide,实现时按实际可用名确认) |
|---|---|---|
| 文档 | txt / md / markdown / mdx | FileText |
| 代码 | ts / js / py / go / rs / java / c / cs / cpp / vue / ... | FileCode |
| 配置 | json / jsonc / yaml / yml / toml / ini / xml / csv / tsv / env / properties | FileCog |
| 资产/工程(Unity 等) | meta / asset / prefab / unity | FileBox |
| 可执行 | exe / bat / cmd / ps1 / sh / msi / app / deb | Cpu |
| 图片 | png / jpg / gif / svg / webp / ico / bmp | FileImage |
| 压缩 | zip / tar / gz / rar / 7z / tgz | FileArchive |
| 锁文件 | package-lock.json / yarn.lock / .env | FileLock |
| 默认(未命中) | 其余 | File |

实现时复用 `file-kind.ts` 已有的扩展名集合(TEXT_EXT 等)做命中判定,不重复维护第二份后缀表;仅把命中结果映射到 icon 名。图标按 lucide-react 实际导出名确认,未命中的图标名回退到 `File`。

**风险**:低。纯增量,icon 传错最坏只是图标不对,不影响功能。

### 3.2 需求 2:文件面板顶部 root 切换

> **待确认**:"当前目录"=`currentCwd` root,"临时工作区"=`MARINA_WORKSPACE` root。若理解有出入请指正。

**改动文件**:
| 文件 | 改动 |
|---|---|
| `FileTreePanel.tsx` | 根部从 `roots.map(section)` 改为:顶部 toolbar(每个 available root 一个按钮)+ 单个 active root 的树 |
| `global.css` | 抽出共享 `.panel-toolbar`/`.panel-toolbar-btn`(从 Git 的 `.git-panel-toolbar` 提炼,两面板复用) |
| 活跃 root 选择 | 走 `usePanelPreference('file-tree','activeRootId')`(L2,跨重启) |
| `GitPanel.tsx`(可选) | toolbar 样式收编到共享类,保持视觉统一 |

**设计决策(经代码核实 `getRoots`/`resolveRoot` 逻辑)**:
- **常态:每个本地终端 = 当前目录 + 临时工作区,两个 root 都可用** → toolbar 永远显示「当前目录 / 临时工作区」两个切换按钮。这是用户的日常体验,与开发者预期一致。
- **单 root / 零 root 只是异常兜底**(非常态),来源:`resolveRoot` 在以下情况标 `available=false`:
  - SSH 会话 → 两个 root 都不可用(marina 不提供 SFTP,文件面板整体无内容)
  - 临时工作区正在创建 / 已随 session 关闭被回收
  - 当前目录被删 / 无权限
- **异常兜底 UI**:只剩一个可用 root 时倾向于「隐藏 toolbar、直接显示那个」;零可用时显示不可用提示(现状已有)。影响极小,可在实现时定。
- **搜索态**:保持现有"跨所有 root 扁平搜索"行为不变(toolbar 隐藏,因为搜索本来就是跨 root 的)。
- toolbar 切换**只切显示哪个 root 的树**,不改数据源、不改懒加载逻辑。

**风险**:中。改的是 FileTreePanel 根渲染,要保证现有"双 root 各自懒加载/搜索/右键"行为不破。需人工回归。

### 3.3 需求 3:UI 状态持久化(系统性)

**基础设施**(见 §2.1):
- 新建 `src/shared/panel-ui-cache.ts` + 单测
- 新建 `src/shared/panel-preferences.ts` + 单测

**落地接入**:
| 面板/状态 | 层 | 落地 |
|---|---|---|
| FileTreePanel 展开目录(`directories`) | L1 | `usePanelUiState` 替换裸 `useState` |
| GitTree 收起目录(`collapsed`) | L1 | 同上 |
| GitPanel viewMode | L2 | 从裸 localStorage 收编到 `usePanelPreference` |
| FileTreePanel activeRootId(需求 2) | L2 | `usePanelPreference` |
| Sidebar expandedPathIds | L1(可选) | 已在 store(跨 mount 保),是否升 L2 跨重启**待确认** |

**范围边界**(防过度扩张):
- 本轮必做:FileTreePanel 展开态、GitTree 收起态(这俩是用户痛点)。
- 顺手做:GitPanel viewMode 收编(零成本,统一规范)。
- **待确认**:Sidebar 展开态是否要跨重启(升 L2)。涉及面广,建议单列。

**风险**:低-中。`usePanelUiState` 是对 `useState` 的薄封装,失败模式退化为"和现在一样丢状态",不会更糟。关键测试:切面板再切回,展开态仍在。

### 3.4 需求 4:缩进统一(系统性)

**改动文件**:
| 文件 | 改动 |
|---|---|
| `global.css` | `:root` 加 `--tree-indent-unit:14px`;`.file-tree-entry .file-tree-entry` 层叠规则**删除**;`.git-panel-group-entries` 的 `margin-left:14px` 改用变量 |
| `FileListRow.tsx` | `marginLeft: depth*14` → `calc(var(--tree-indent-unit) * depth)` |
| `FileTreePanel.tsx` | `DirectoryChildren`/`FileTreeEntryRow` 加 `depth` prop,递归传递(根 0,每层 +1),传给 `FileListRow` |

**风险**:低。FileTree 当前视觉就是 14px/层,改完像素不变(只是机制从 CSS 层叠换成 depth prop)。关键测试:多级嵌套目录缩进与改前一致、与 Git 树一致。

## 5. 风险与权衡

| 风险 | 等级 | 应对 |
|---|---|---|
| 需求 2 改 FileTreePanel 根渲染破坏双 root 行为 | 中 | 人工回归:懒加载、搜索、右键、不可用 root 提示 |
| `usePanelUiState` 封装有隐蔽 bug 导致状态错乱 | 低-中 | 薄封装 + 单测;失败退化 = 现状(不更糟) |
| PanelPreferences 迁移老 localStorage key 漏掉 | 低 | 迁移函数 + 单测覆盖三个老 key |
| 缩进改造引入像素级偏差 | 低 | depth 机制与原 14px 等价;逐主题目测 |
| 扩展名→icon 映射覆盖不全 | 低 | 不全则回退默认 `File`,无害;可后续扩 |
| scope 蔓延(sidebar 等全量持久化) | 中 | §3.3 明确边界,sidebar 跨重启单列待确认 |

**不做的事**(防过度设计):
- 不引入状态管理库(Zustand/Redux)。现有 `useReducer+Context`(store.tsx) + 模块级缓存够用,加库违反 AGENTS.md 边界 2。
- 不改 `LayoutHost` 的 `<ActivePanel key>` lazy mount。需求 3 用组件外缓存解决,不靠改 mount 机制。
- 不做面板状态的 main 端持久化(那会引入 IPC + 磁盘 IO,UI 态不值得)。
