# Marina 打包发布流程

**最后更新**:2026-05-20
**适用版本**:0.1.0-beta.10 起
**作者**:Liyue-Cheng + Claude Opus 4.7
**文档状态**:**当前唯一权威**(替代之前散落在 `docs/checkpoints/` / `AGENTS.md` 4.2 / `CHANGELOG.md` 里的零散描述)

> 历史背景:beta.9 之前每次发版都是作者手工跑一串命令,任何漏掉或顺序错都会带来回归。最直接的代价是 ISO-1(详见 `docs/issues/iso-1-cross-platform-build-pollution.md`)— 从 beta.5 到 beta.9 共 5 个安装包都夹带错平台 .node 二进制,没人系统校验过。本文档把整套流程钉死,让 release 从"作者本地经验"变成"任何人在任何机器上能复现"。

---

## 0. 一句话:怎么发版?

```bash
# 单平台
npm run release:win               # Windows 包(nsis + portable)
npm run release:linux             # Linux 包(.deb + .rpm + .AppImage)

# 全平台一次发完
npm run release:all

# 升版号 + 发版一气呵成
npm run release:all -- --bump=prerelease

# 不知道会跑什么?先看一下
npm run release:win -- --dry-run
```

完整选项见 `npm run release -- --help`。

## 1. 整体设计

### 1.1 流水线六阶段

```
┌────────────────────────────────────────────────────────────┐
│                    npm run release:* [选项]                 │
├────────────────────────────────────────────────────────────┤
│                                                            │
│   Phase 1 · Preflight                                      │
│   ├─ Node ≥ 20                                             │
│   ├─ git status / 当前分支                                  │
│   ├─ 平台兼容(Win 主机不能直跑 Linux 包,自动走 Docker)    │
│   └─ --bump=*  → 改 package.json.version                   │
│                                                            │
│   Phase 2 · Code Quality                                   │
│   ├─ typecheck(三 tsconfig)                                │
│   ├─ ESLint + stylelint                                    │
│   └─ vitest run                                            │
│                                                            │
│   Phase 3 · Clean & Switch                                 │
│   ├─ clean out/ + node-pty/build(避免跨平台残留)           │
│   └─ install-app-deps --platform=<target>                  │
│      ↑ Windows 主机打 Linux 包时跳过(Docker 容器内 npm ci) │
│                                                            │
│   Phase 4 · Build                                          │
│   ├─ Windows: electron-vite build + electron-builder       │
│   └─ Linux:                                                │
│      ├─ Linux 主机: 直跑 npm run build:linux                │
│      └─ Win/Mac 主机: 走 Dockerfile.linux-build 隔离构建    │
│                                                            │
│   Phase 5 · Verify                                         │
│   ├─ 扫所有 .node 文件,按 PE/ELF/Mach-O magic 判平台        │
│   ├─ 关键资源在位(app.asar / shell-hooks/ / context-menu)│
│   ├─ 体积合理(80 MB ≤ unpacked ≤ 500 MB)                 │
│   └─ 输出每个安装器的 SHA256                                │
│                                                            │
│   Phase 6 · Report                                         │
│   ├─ 产物清单 + 大小                                       │
│   └─ 下一步:CHANGELOG / commit / tag / push / GH release   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 1.2 核心设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 哪个脚本是入口? | `scripts/release.mjs`,单文件总流水 | 用户不需要记 12 条命令,记一条 `release` 就够。每条 npm script 不超过一行 |
| 脚本语言? | Node.js `.mjs`,无新增依赖 | 跨 Win/Linux/Mac 一致;`node:fs` / `child_process` 够用,不引 `rimraf` / `cross-env` / `semver` 等(避免 AGENTS.md 1.2 边界 2) |
| 平台不在主机时怎么办? | Win 主机 → Linux 走 Docker;Linux 主机 → Win 不支持(本地报错) | Docker 隔离已经证明能根治 ISO-1;反向 Win-on-Linux 需要 wine,留给 CI |
| 失败时怎么办? | 任何阶段非零立即退出,产物保留供检查 | 不做"自动重试"或"自动清理失败产物"— 这两个都会掩盖根因 |
| 跳过项的安全网? | 每个 `--skip-*` 都打黄色警告;`--skip-verify` 额外打红字 | 跳过是诊断用的,不是常态。让作者每次跳过都看到警告 |

## 2. 三层防御回顾(为什么有 verify 这步)

ISO-1 工单的核心教训是:**electron-builder 没报错 ≠ 产物没问题**。打包过程的成功只意味着"我把文件塞进了 .exe / .deb",不意味着"塞进去的文件平台正确"。

所以本流程在三个不同层级各设防御:

```
┌────────────────────────────────────────────────────────┐
│                    源头:node_modules                   │
│   build/Release/pty.node 跨 Win/Linux 共用同一路径,     │
│   每次切平台都会被覆盖                                   │
│                                                        │
│   防御 1 (Phase 3):清 native + install-app-deps        │
│      --platform=<target>  目标平台 ABI 编译             │
└────────────────────────────────────────────────────────┘
                            ↓
┌────────────────────────────────────────────────────────┐
│             中游:electron-builder staging               │
│   默认把 node-pty 整个子树打进每个包                     │
│                                                        │
│   防御 2 (electron-builder.yml):per-platform files       │
│      负 glob,在 staging 阶段就剔掉错平台二进制          │
└────────────────────────────────────────────────────────┘
                            ↓
┌────────────────────────────────────────────────────────┐
│                  下游:打包后的 .exe / .deb             │
│   即使前两层都失误也别让用户拿到坏包                     │
│                                                        │
│   防御 3 (Phase 5):verify-artifacts.mjs 扫所有 .node    │
│      用 magic bytes 判平台,不匹配 = 立即失败            │
└────────────────────────────────────────────────────────┘
```

三层独立,任一层失效另两层兜得住。详见 `docs/issues/iso-1-cross-platform-build-pollution.md`。

## 3. 各阶段细节

### 3.1 Phase 1 · Preflight

**做什么**:
- 读 `package.json` 拿当前版本
- 检查 Node 版本 ≥ 20(Electron 31 硬要求)
- `git status --porcelain`:有未提交改动时打黄字警告(不阻塞)
- `git branch --show-current`:显示当前分支
- 如果有 `--bump=*`:计算新版本号并写入 `package.json`

**版本号策略**(`--bump`):

```
当前 0.1.0-beta.9 + --bump=patch       → 0.1.1-beta.0
当前 0.1.0-beta.9 + --bump=minor       → 0.2.0-beta.0
当前 0.1.0-beta.9 + --bump=major       → 1.0.0-beta.0
当前 0.1.0-beta.9 + --bump=prerelease  → 0.1.0-beta.10
当前 0.1.0       + --bump=prerelease   → 0.1.1-beta.0
当前 0.1.0-beta.9 + --bump=none(默认)→ 0.1.0-beta.9
```

**手工版本号修改**也支持 — 直接编辑 `package.json.version`,然后跑 `npm run release` 不带 `--bump`。

**不做什么**:
- ❌ **不自动 commit**:`--bump` 只改 `package.json`,不 `git add`、不 `git commit`。CHANGELOG 是 prose,自动化反而容易写错
- ❌ **不阻塞 dirty git**:让作者自己决定。文档里反复警告但不卡

### 3.2 Phase 2 · Code Quality

**做什么**:
- `npm run typecheck`(三 tsconfig 串行)
- `npm run lint`(ESLint)
- `npm run lint:css`(stylelint)
- `npm test`(vitest run)

**跳过策略**:
- `--skip-typecheck` / `--skip-lint` / `--skip-tests` 各自独立
- 黄字警告,但不阻塞

**已知失败处理**:
- 截至 2026-05-20,`css-var-fallback.test.ts` 因 Catppuccin 主题 CSS 变量遗漏 fallback 而失败
- 修这个的工单单独走,**不**用 `--skip-tests` 当成长期解决方案

### 3.3 Phase 3 · Clean & Switch

**做什么**:
- `node scripts/clean.mjs --out --native`:清 `out/` 与 `node_modules/node-pty/build/`
- 按目标平台跑 `electron-builder install-app-deps --platform=<target> --arch=x64`

**矩阵**:

| 目标 \ 主机 | Windows | Linux | macOS |
|---|---|---|---|
| `--win` | switch:win | 报错(需 wine) | 报错 |
| `--linux` | 跳过(Docker 自己 ci) | switch:linux | 跳过(Docker) |

**为什么 Windows 主机打 Linux 包要跳过 switch?**
- Linux 包在 Docker 容器内构建,容器有自己的 `node_modules`(来自 `npm ci`)
- 主机的 `node_modules` 不进容器(`.dockerignore` 拦)
- 切平台只对"主机直接 electron-builder"路径有意义

### 3.4 Phase 4 · Build

**Windows 路径**:
```
electron-vite build → out/{main,preload,renderer}/
electron-builder --win --x64 → release/${version}/{Marina-Setup.exe, Marina-Portable.exe, win-unpacked/, latest.yml, *.blockmap}
```

**Linux 路径(主机是 Linux)**:
```
electron-vite build → out/...
electron-builder --linux deb rpm AppImage --x64 → release/${version}/{Marina-*.deb, *.rpm, *.AppImage, linux-unpacked/, latest-linux.yml}
```

**Linux 路径(主机非 Linux)**:
```
node scripts/build-linux-docker.mjs:
  1. docker build -f Dockerfile.linux-build -t marina-linux-build:local .
  2. docker run --rm -v "${PWD}/release:/build/release" marina-linux-build:local
     ↑ 容器内 CMD = npm run build:linux
```

详见 `Dockerfile.linux-build` 顶部注释 + `docs/issues/iso-1-cross-platform-build-pollution.md` §"修复 3"。

### 3.5 Phase 5 · Verify

**检查项**:

1. **二进制平台一致性**
   - 扫 `release/${version}/{win,linux}-unpacked/` 下所有 `*.node`
   - 读前 16 字节判 PE / ELF / Mach-O / FAT
   - Windows 包不能含 ELF 或 Mach-O;Linux 包不能含 PE 或 Mach-O;macOS 包不能含 PE 或 ELF

2. **关键资源在位**
   - `resources/app.asar` 文件
   - `resources/shell-hooks/` 目录
   - Windows 额外:`resources/context-menu/` 目录(MSIX 等)

3. **体积合理**
   - 下限 80 MB(< 这个数说明产物缺失)
   - 上限 500 MB(> 这个数说明漏过滤)
   - `--strict` 模式下超上限算失败,默认算警告

4. **安装器输出**
   - 每个 `Marina-*.exe / .deb / .rpm / .AppImage` 的大小 + SHA256
   - `.blockmap` 跳过 hash(它本身就是 hash 文件)

**机器可读输出**:
```
node scripts/verify-artifacts.mjs --json > verify-report.json
```

JSON 含每个包的完整 `nodeBinaries` 数组,可被 CI 解析。

### 3.6 Phase 6 · Report

**输出**:
- 各阶段耗时
- 产物清单与大小
- 下一步提示(仅在 `--bump=*` 时显示)
  - 改 CHANGELOG.md
  - git add + commit + tag + push
  - 在 GitHub Releases 上传

## 4. CI 化路径(未来)

本流程**为 CI 而设计**,但当前未启用 GitHub Actions。要 CI 化只需:

```yaml
# .github/workflows/release.yml
name: Release
on:
  push:
    tags: ['v*']
jobs:
  windows:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: npm run release:win -- --strict
      - uses: actions/upload-artifact@v4
        with:
          name: marina-windows
          path: release/${{ github.ref_name }}/Marina-*.exe

  linux:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: npm run release:linux -- --strict
      - uses: actions/upload-artifact@v4
        with:
          name: marina-linux
          path: release/${{ github.ref_name }}/Marina-*.{deb,rpm,AppImage}
```

不需要 macOS runner 直到 V2.0 启用 mac 包。

**未启用 CI 的原因**:
- 代码签名证书(KI-002 / KI-003 共享)还没买,Windows 包签名无证书无意义
- alpha/beta 阶段每次发版要观察用户反馈,作者本地走完整流程更稳

签名证书到位 + 公测前一周启用 CI,见 `docs/known-issues.md` KI-002 §"推荐分阶段"。

## 5. 故障排查

### 5.1 Phase 2 卡在 typecheck

**症状**:`tsc --noEmit` 报错。

**通常原因**:
- TypeScript 版本升级后语法变化
- 三个 tsconfig 中某个的 `include` / `exclude` 漏了

**修法**:跑 `npx tsc --noEmit -p tsconfig.node.json`(或 preload / web)逐个看;不要直接 `--skip-typecheck`。

### 5.2 Phase 3 install-app-deps 卡很久

**症状**:`electron-builder install-app-deps` 长时间无输出,最终 fail。

**通常原因**:
- 国内网络拉 Electron 二进制慢 / 失败
- node-gyp 找不到 Python / Visual Studio Build Tools(Windows)
- node-gyp 找不到 gcc / make(Linux,不会发生因为 Docker 镜像已装)

**修法**:
- 设镜像:`set ELECTRON_MIRROR=https://npmmirror.com/mirrors/electron/`(Win)/ `export ...`(Linux)
- Windows 上确保装了 Visual Studio Build Tools 2022 + Python 3.10+

### 5.3 Phase 4 Docker build 失败

**症状**:`docker build` 在 `RUN npm ci` 步骤 fail。

**通常原因**:
- Docker daemon 没启动
- `package-lock.json` 与 `package.json` 不同步
- 主机 `node_modules` 进了容器(`.dockerignore` 没拦住)

**修法**:
- `docker info` 检查 daemon
- 主机跑一次 `npm install` 同步 lockfile
- 看 `.dockerignore` 是否在仓库根(不是子目录)

### 5.4 Phase 5 verify 报"cross-platform-binary"

**症状**:`verify-artifacts.mjs` 输出 `✗ [cross-platform-binary] ...node 是 ELF (Linux),但 win 包应该是 PE (Windows)`。

**通常原因**:
- `electron-builder.yml` 的 `files` 过滤规则被改动
- `node_modules/node-pty/build/Release/pty.node` 来自上次错平台构建
- `asarUnpack` 通配符太宽

**修法**:
- 看 `electron-builder.yml` 是否还有 ISO-1 引入的 `files` 负 glob 块
- 跑 `npm run clean:native` + `npm run switch:<target>` 重置 native
- 重打:`npm run release:<target>`

### 5.5 Phase 5 报"size-too-small"

**症状**:unpacked 体积 < 80 MB。

**通常原因**:
- `electron-vite build` 失败但 `electron-builder` 仍然跑了
- `out/` 在 build 之前没干净,只 build 了一部分

**修法**:`npm run clean` + `npm run release:<target>`(不带 `--no-clean`)。

### 5.6 verify 通过了但用户跑安装器仍报 .node 加载失败

**症状**:`Cannot find module ... .node` 或类似。

**通常原因**:
- node-pty `prebuilds/` 被过滤太狠(把当前平台也排除了)
- Electron ABI 不匹配(install-app-deps 跑成了 plain Node ABI 而不是 Electron's)

**修法**:
- 看 `electron-builder.yml` 当前平台 `files` 块,确认没把目标平台 prebuilds 也排除
- 删 `node_modules/`,从头 `npm install`(重跑 postinstall = install-app-deps)
- 看 `release/${version}/<plat>-unpacked/resources/app.asar.unpacked/node_modules/node-pty/` 实际有哪些 .node

## 7. 文件清单

本流程依赖以下文件,任一损坏需补全:

```
package.json                              ← scripts.release* + verify
scripts/release.mjs                       ← Phase 1-6 总入口
scripts/verify-artifacts.mjs              ← Phase 5
scripts/clean.mjs                         ← Phase 3 与日常清理
scripts/build-linux-docker.mjs            ← Phase 4 Linux on Win/Mac 路径
Dockerfile.linux-build                    ← Docker 镜像定义
.dockerignore                             ← 阻断主机 node_modules 污染
electron-builder.yml                      ← 平台 files 过滤(ISO-1)
```

测试:不依赖测试夹具,但 `npm test` 必须通过(或用 `--skip-tests` 显式跳过,需配单独工单解决根因)。
