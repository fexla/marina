# 0.3.2 需求感知后台任务调度方案

**状态**：已实现，待开发者手动验收  
**目标版本**：0.3.2（开发期保持 `package.json` 0.3.1，记入 `[Unreleased]`）  
**日期**：2026-07-22

## 1. 背景

0.3.1 修复了 Git watcher 在 Session `exited` / `destroyed` / 离开仓库 / 禁用 Git 后仍继续运行，以及慢 `git status` 重叠的问题。但该修复仍是 GitService 内部固定 `setInterval(3000)`：只要 Session 还活着且 cwd 在仓库内，无论用户当前看哪个终端、哪个面板、Marina 是否失焦或处于零窗口托盘态，都会每 3 秒 spawn `git.exe`。

这会让“生命周期正确”仍不等于“资源投入与用户需求匹配”。本方案增加一个仅面向**昂贵周期后台工作**的需求感知调度器，并首先迁移 Git status。

## 2. 设计目标

1. 当前聚焦窗口中，当前 Session 的 Git 面板可见时，保持 3 秒新鲜度。
2. 当前 Session 仍显示但 Git 面板隐藏、dock 折叠、窗口失焦或页面隐藏时，降为 60 秒。
3. Session 不再显示、owner 释放、Session 退出、Git 不可用或没有任何客户端需求时，完全停止周期扫描。
4. WARM/COLD 切回 HOT 时立即刷新，不等待下一周期。
5. 任务完成后才安排下一次；任何任务实例永不重叠。
6. 所有昂贵后台任务共享全局并发预算，v1 默认同时最多运行 1 个。
7. 本地 Electron IPC 与远程 WS 后端使用同一 handler / clientId 语义。
8. 窗口关闭、远程断线、owner 变化和 Session 销毁都能清理 demand，不留下幽灵任务。
9. 不新增依赖；调度器不记录 task key、sessionId 或路径到自动性能报告。

## 3. 明确不做

- 不把所有 timer 统一塞入调度器。event-loop stall detector、Session idle timeout、持久化 debounce 等是语义时钟，不应随 UI 可见性改变。
- 不取消已经启动的 `git.exe`。已有 5 秒 timeout；需求降级只阻止后续调度。
- 本期不合并同仓库多 Session 的 status 任务；报告数据稳定后再评估按 repoRoot 合并。
- 不承诺精准到毫秒的周期；3 秒/60 秒表示“上一次完成后至少等待该时长”。
- 不让 renderer 传任意 task 名；协议只暴露枚举化 Git demand 命令。

## 4. Demand 状态模型

```text
             Git visible + focused
       ┌─────────────────────────────┐
       │                             ▼
    COLD ───── selected/displayed ─ WARM ──────> HOT
      ▲              but hidden       ▲           │
      │                               │           │ blur / panel switch
      └── unmount / owner lost ───────┴───────────┘
```

| 等级 | 条件 | 调度 |
|---|---|---|
| `hot` | Git tab 活跃、dock 未折叠、document visible、窗口有焦点 | 立即跑一次；完成后 3 秒再跑 |
| `warm` | 当前 Session 的 PanelStack 仍挂载，但 Git 不可见/窗口失焦 | 完成后 60 秒再跑；进入 WARM 不强制立即跑 |
| `none` | PanelStack 卸载、Session 换走、owner 释放、退出/销毁 | 清 timer；运行中的本次允许完成但不再续排 |

同一 task 可有多个 consumer（多窗口/远程 client）；有效等级取最高：`hot > warm > none`。

## 5. Renderer 信号

`LayoutHost.PanelStack` 是“当前可显示 Session + 当前 dock 页面”的单一真相源。新增 hook：

```typescript
useGitPollingDemand({
  sessionId,
  gitAvailable: panelIds.includes('git'),
  gitVisible: activePanelId === 'git' && !persisted.collapsed,
  isOwner: session.ownerWindowId === appState.myWindowId,
});
```

hook 监听：

- `activePanelId`；
- dock `collapsed`；
- `session.id`；
- `window` focus/blur；
- `document.visibilitychange`。

每次状态变化发送绝对状态：

```typescript
cmd:git:set-polling-demand
{ sessionId, level: 'hot' | 'warm' | 'none' }
```

非 owner 一律上报 `none`；`ownerWindowId` 必须进入 hook 依赖，解决 claim/release 后 PanelStack 未 remount 的情况。cleanup 发送 `none`。发送失败只记 `console.warn`，不能影响 UI。命令属于 `backend-data`：远程窗口必须把需求发送到它当前连接的 daemon，而不是本地客户端 main。

Demand 可能早于 `prefetchStatus()` 完成与 task 注册到达。Scheduler 必须有有界的 pre-registration demand：未注册 key 的绝对 demand 暂存；`registerTask` 立即吸收并按当前最高等级调度；`unregisterTask`、Session 生命周期清理和 consumer 清理同时删除 placeholder，不能丢首次 HOT。

## 6. Main 安全边界

handler 从 `CommandEnvelope.windowId` 取得 consumerId，renderer 无权自报 consumer：

```typescript
gitService.setPollingDemand(sessionId, envelope.windowId, level)
```

`hot` / `warm` 必须验证：

- Session 存在；
- Session 非 `exited`；
- `session.ownerWindowId === consumerId`；
- Git 功能当前启用。

`none` 始终幂等允许，以便 Session 已销毁后仍能清理。

owner 变化时 GitService 清空该 Session 的旧 demand；新 owner 的 renderer mount 后重新上报。窗口关闭调用 `removeConsumer(windowId)`；远程连接断开也调用相同清理，不等待 Session owner 的重连宽限期。重连后 renderer 会重新上报绝对状态。

## 7. BackgroundWorkScheduler

新增 `src/main/background-work-scheduler.ts`，只依赖 Node timer 和固定名称性能指标。

### 7.1 API

```typescript
type BackgroundDemandLevel = 'none' | 'warm' | 'hot';

interface BackgroundTaskDefinition {
  run: () => Promise<void>;
  hotIntervalMs: number;
  warmIntervalMs: number;
}

registerTask(key, definition): void;
unregisterTask(key): void;
setDemand(key, consumerId, level): void;
clearTaskDemands(key): void;
removeConsumer(consumerId): void;
shutdown(): void;
```

Task key 只在 main 内部使用，例如 `git-status:<sessionId>`，不得进入日志、IPC 响应或自动报告。

### 7.2 调度算法

- 使用递归 `setTimeout`，禁止 `setInterval`。
- `none → hot`、`warm → hot`：取消旧 timer，立即入队。
- `hot → warm`：取消 3 秒 timer，从切换时刻重新等待 60 秒。
- `* → none`：取消 timer；若正在运行，仅标记完成后不续排。
- run 完成后读取**最新**有效 demand，再决定下次间隔。
- `queued` / `running` 两个 flag 保证同一 task 只入队一次。
- 队列使用 insertion-ordered `Set<TaskRecord>`，物理去重/删除且天然受 task 上限约束；降为 WARM/NONE、unregister 或同 key 重注册会让旧 queued/run completion 失效，旧任务绝不能执行或给新任务续排。
- 全局 FIFO + `maxConcurrent=1`，避免多个仓库同一时刻 spawn 多个 `git.exe`。
- timer `unref()`，调度器不单独阻止应用退出。
- task 抛错由 scheduler 捕获并调用固定 `onError`，队列继续；业务错误不得形成 unhandled rejection。

### 7.3 有界性

- task 数由活 Session 数上限约束；Git 离仓/退出立即 unregister。
- consumer demand 由窗口/client 数与 task 数笛卡尔积约束；窗口/断线统一清理。
- pending queue 是 Set，不允许重复 key。
- shutdown 清所有 timer/task/demand/queue。

## 8. GitService 迁移

现有 `watchers` Map 保留为“已注册 polling task”的兼容注册表，但 value 不再是 `setInterval`；其 `close()` 调 scheduler.unregisterTask。

单次 poll 逻辑继续复用 `emitCurrentStatus()`，已有 `pollsInFlight` 作为纵深防御。迁移后：

```text
prefetchStatus available
  → 只注册 git task（初始 COLD，不无条件 spawn）
  → renderer demand 到达
      → HOT 立即 / WARM 60s 调度

GitPanel mount
  → 交互 getStatus + HOT immediate 同 cwd 合并
  → 全局根层 cache bridge 持续接收 WARM/HOT 事件

cd out / disable / exited / destroyed
  → unregister task + clear task demands
```

`getStatus` 仍是交互请求，不等待后台 FIFO；GitPanel mount 的 `GIT_GET_STATUS` 与 HOT immediate 共享 per-session + cwd in-flight coalescer。同 cwd 配置 revision 即使变化也先等旧进程结束，再串行重拉，任何时刻只跑一份；cwd 已变化时旧结果不得覆盖新状态。

`prefetchStatus` 不再无条件执行昂贵 status，因此多个 Session 同时变 available 不会绕过 scheduler 并发预算。WARM 时 GitPanel 已卸载，`App.ConnectedShell` 的常驻 `GIT_STATUS_UPDATED` bridge 负责写模块级 cache；切回 Git 先秒显 WARM 更新过的缓存，再由 HOT/getStatus 补真值。

## 9. 生命周期与远程连接

### 本地窗口

```text
WindowManager closed
  → gitService.removePollingConsumer(windowId)
  → SessionManager.handleWindowClosed(windowId)
```

先清 demand，再 release owner；两条路径都幂等。

### 远程 client

RemoteDaemonController 把 `onClientGone(clientId)` 透传给 GitService：

```text
WS disconnect
  → removePollingConsumer(clientId) 立即降级
  → owner 保留重连宽限期（既有）
  → 重连后 renderer 重新上报 demand
```

controller stop/restart 会在清 socket mapping 前显式触发每个 client 的 disconnected，
不能依赖异步 close 恰好回调；close identity guard 防止重复清理。

这避免网络断开后 10 秒宽限期内仍维持 HOT poll。

### 应用退出

`before-quit` 同步调用 scheduler shutdown，正在执行的 git 进程继续受既有 timeout/进程退出控制；不再安排下一轮。

`prefetchStatus` 另维护 per-session availability epoch。每次可用性 flip/prefetch、disable、exit、destroy 都递增 epoch；异步边界后必须同时复核 epoch、cwd、enabled 和 Session state。旧 available 结果不得在较新的 cd-out/disable 之后重新注册 task，也不得 emit 陈旧 snapshot。

## 10. 性能指标

只记录固定名称：

- counter：registered / unregistered / demandChanges / runs / runErrors / queueDeduplicated；
- gauge：tasks / hotTasks / warmTasks / running / queued；
- Git 既有 watchers / pollsInFlight / skipped 继续保留。

不记录 task key、consumerId、sessionId、repoRoot。

## 11. 测试矩阵

### Scheduler 单元测试

- demand 早于 register 时不丢，register 后按最新 demand 调度；
- `none` 不运行；
- HOT 立即运行，完成后 3 秒；
- WARM 首次等待 60 秒；
- WARM→HOT 立即；HOT→WARM 取消旧 3 秒 timer；
- HOT→none 运行完成后不续排；
- 同 task 永不重叠；
- 两 task 在 `maxConcurrent=1` 下串行；
- 多 consumer 取最高 demand；移除 consumer 正确降级；
- queued 后降级/注销不执行；同 key unregister→register 后旧 completion 不影响新 record；
- unregister/shutdown 清 timer/queue；
- run reject 不阻塞后续任务。

### Git/IPC/Renderer 测试

- demand owner 校验、`none` 幂等；
- available 注册 task，离仓/disable/exited/destroyed 注销；
- owner changed 清旧 demand，owner false→true 即使 PanelStack 不 remount 也重新上报；
- window close / remote disconnect 清 consumer；
- 新 IPC 注册且保持 backend-data；
- LayoutHost hook 的 tab/owner/focus/visibility/unmount 由代码审阅 + 隔离 Electron runtime smoke 验证（遵循项目“不写 renderer UI 自动测试”规则）；
- GitPanel mount getStatus 与 HOT immediate 同 cwd 合并，配置变化只串行重拉；
- available prefetch 无 UI demand 时不 spawn；WARM 事件由根层 cache bridge 持续接收；
- availability epoch 阻止旧 prefetch 在 cd-out/disable 后复活；
- 既有慢 poll 防重叠与 disable/exit 竞态继续通过。

## 12. 验收标准

1. 聚焦 Git 面板：立即刷新，随后约每 3 秒。
2. 切到文件/已打开面板或 Marina 失焦：降为每 60 秒。
3. 切到其他 Session 或关闭全部窗口：旧 Session 不再周期 spawn Git。
4. 切回 Git：不等待 60 秒，立即刷新。
5. 多 Session 同时到期时最多一个后台 `git status` 在跑。
6. 本地与远程 backend 行为一致。
7. typecheck、lint、lint:css、全量测试、build 通过；版本仍为 0.3.1，CHANGELOG 记 `[Unreleased]`。
