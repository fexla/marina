### 1.2 核心状态机(RemoteTransport)

```
new RemoteTransport
  │
  ▼ connectWs(true) [首次:设握手超时]
┌─────────────┐
│  connecting  │ ← ws.onopen → 发 auth 帧(clientId=null 不带 resume)
└──────┬──────┘
       │ 收 __auth-ok__ { clientId }
       ▼
┌─────────────┐
│   ready      │ ← invoke/on 可用
└──────┬──────┘
       │ ws.onclose(网络断)
       ▼
┌─────────────┐  reject pending invoke
│ reconnecting │ ← invoke reject("重连中")
│  (backoff)   │  onReconnectStart
└──────┬──────┘
       │ setTimeout(min(base·2^attempt, 30s))
       ▼ connectWs(false) [重连:不设握手超时]
┌─────────────┐
│  connecting  │ ← ws.onopen → 发 auth 帧(带 resumeClientId=原 clientId)
└──────┬──────┘
       │ 收 __auth-ok__ { clientId(复用同值) }
       ▼
┌─────────────┐
│   ready      │ ← onReconnectSuccess → preload reload 重拉 snapshot
└─────────────┘

任意状态 ──close()──► closed(manuallyClosed 终态,不重连)
reconnecting 达 maxReconnectAttempts(默认10) ──► closed + onReconnectFail
```

**握手前断线**(clientId=null 且非重连)= 初始化失败,failReady,**不重连**(初始化失败由上层报错;只有"曾成功连过"才重连)。

### 1.3 daemon 端 clientId 复用(V1 单 client 模型)

```
RemoteDaemon.boundClientId: string | null  (首次握手分配并存)

makeAuthHandler(firstMessage):
  parseAuthFrame → { type:'auth', token, resumeClientId? }
  if token !== deps.token → error('token mismatch')
  if resumeClientId && resumeClientId === boundClientId → 复用 boundClientId
  else → 分配新 randomUUID + 更新 boundClientId
```

V1 场景:单 client + 单 token,daemon 记一个 boundClientId 即可。多 client(V2)需扩展为 token→clientId 映射。

### 1.4 自动 release(阶段1 实现,本阶段确认)

```
ws.on('close')  [transport-ws.ts L293]
  → disconnectedHandlers(clientId)
  → remote-daemon onClientDisconnected(clientId)
  → registry.remove(clientId)
  → sessionManager.handleWindowClosed(clientId)  // 该 client 的 session owner=null
  → onClientGone?.(clientId)
```

session **不销毁**,scrollback 保留(2MB 环形缓冲)。重连后 client reload → GET_SNAPSHOT 看到这些 session(owner=null),用户点击即 claim。

### 3.2 已知限制 / 未测

- **真实拔网线 / sleep-wake 未自动测**:这些是 OS 级事件,需人工触发(见用户测试指南)。自动测试用 `fireClose()` 模拟 WS 断开,覆盖逻辑路径,但不覆盖真实网络中断的时序细节(如 TCP TIME_WAIT、Wi-Fi 重连延迟)。
- **重连达上限(10 次 × 指数 backoff ≈ 15 分钟)未自动测**:耗时长,且逻辑简单(attempt 计数 + onReconnectFail)。代码路径被"主动 close 不重连"间接覆盖。
- **UI 无"重连中"视觉提示**:onReconnectStart 只 console.warn,reload 前 UI 会卡(invoke 全 reject)。留待后续加 banner(类似阶段2 review 的 UX follow-up #4)。
- **重连后 session owner 全 null**:reload 后用户看到 session 列表但需手动点 claim。未自动 re-claim 断线前 focused 的 session(V1 简化:断线是异常,用户重新介入合理)。

## 4. 用户测试指南(检查点:断 30s 重连,session 还在,scrollback 完整)

### 准备
- 两台机器 A(client)+ B(daemon),WireGuard 直连(或同 LAN)
- B 装好 Marina,`marina --headless --daemon` 启动,托盘"复制 daemon 配对 token"
- A 装好 Marina,设置 → 远程后端 → 添加 profile(B 的 IP:12580 + token)→ "在新窗口打开"
- A 该窗口应显示 B 的 session 列表(远程 snapshot)

### 测试 1:断线 → 自动重连(预计 1 分钟)
1. A 的远程窗口正常:在 B 的某 session 里跑 `for ($i=1;$i -le 100;$i++){sleep 1;Write-Host $i}`(PowerShell,每秒输出一个数)
2. **断 B 的网络**(拔 B 网线 / 关 B 的 WireGuard / B 上 `marina` 停 daemon)
3. **预期(断线瞬间)**:A 远程窗口的 session 停止更新(字节流断),console 有 `[preload] remote connection lost — reconnecting...`
4. **预期(断线期间)**:在 A 远程窗口操作(如点别的 session)会报错(invoke 重连中被拒)
5. **恢复 B 网络**(插回 / 重启 daemon)
6. **预期**:A console 在 backoff 后出现 `[preload] remote reconnected — reload to refresh snapshot`,**窗口自动 reload**
7. reload 后:session 列表回来,刚才那个跑循环的 session **还在**,点进去 scrollback 完整(1-100 的输出都在,期间断的数字 daemon 端 PTY 继续产生,scrollback 环形缓冲保留了)

### 测试 2:daemon 重启(区别于网络断)(预计 1 分钟)
1. A 远程窗口正常连 B
2. **完全重启 B 的 daemon**(`marina` 停 + 重新 `--headless --daemon`)
3. **预期**:B 重启后 token 不变(daemon-credentials 持久化),A 的 RemoteTransport backoff 重连
4. **预期**:重连成功(同 token + boundClientId 复用)。但因 B 重启,session 全没了(daemon 不持久化 session,设计如此),A reload 后看到空列表 —— **这是正确的**(daemon 重启 ≠ 网络断)

### 测试 3:sleep-wake(笔记本合盖)(预计 2 分钟)
1. A 是笔记本,远程窗口正常连 B
2. **合盖 sleep**(不是关机)
3. 等 30 秒
4. **开盖 wake**
5. **预期**:Wi-Fi 重连后,RemoteTransport 检测到 ws 断(sleep 时 TCP 超时)→ backoff 重连 → reload → session 恢复
