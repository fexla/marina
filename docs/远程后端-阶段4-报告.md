## 1. 验证方法与结论

阶段4 与阶段1-3 不同:**它主要是验证,不是新代码**。方案 §IX/§I.2 的核心论点是"Main 业务逻辑整体复用,只动传输层"——如果论点成立,file-panel/SSH/tmux 应**免费工作**(零额外设计)。本次通过代码审查 + 链路追踪验证了这个论点。

### 1.1 file-panel 远程链路 ✅

**两条路径,都通:**

**路径 A:agent 触发打开文件(agent → daemon → client UI)**
```
agent(daemon B 的 PTY 里)
  │ env.MARINA_SERVICE=http://127.0.0.1:<filePanelPort>  [session-manager L711-714 注入]
  │ env.MARINA_TOKEN=...
  │ env.TERMINAL_ID=<sessionId>
  ▼ POST /open-file {terminal, path}
daemon B 的 file-panel-service(HTTP server, L148 start)        ← daemon 模式下照常启动
  │ filePanelService.openFile(sessionId, path)                  ← 读 daemon 本地文件
  │ emit 'filePanelUpdated'
  ▼ ipc.ts wireEventBroadcasts 订阅 → **定向 sendEventTo(session.ownerWindowId)**
  │   (ownerWindowId 在 WS 模式 = client 的 clientId;不是 broadcastToAll)
WS event 帧 → client A 的 RemoteTransport.on → store 更新 → FilePanel 渲染
```

**路径 B:client 操作 panel(client → daemon → client UI)**
```
client A 点 panel 项
  │ RemoteTransport.invoke('cmd:file-panel:show-file', {sessionId, path})
  ▼ WS command 帧
daemon B dispatcher → ipc handler FILE_PANEL_SHOW → filePanelService.showFile
  │ emit 'filePanelUpdated'
  ▼ 广播回 client(同路径 A 后半段)
```

**代码审查证据**:
- `index.ts` L392 `filePanelService.start()` 在 daemon 分支(L549)**之前**执行 → daemon 模式下 file-panel HTTP 照常起
- `session-manager.ts` L711-714:`filePanelService.getUrl()` 返回 daemon 本地 url,注入 agent PTY env。daemon 模式下 agent 跑在 daemon 机器,127.0.0.1 指向 daemon 自己 ✓
- `ipc.ts` L1463-1493:file-panel command(FILE_PANEL_GET_OPEN_FILES / OPEN / SHOW / CLOSE / READ_FILE / READ_IMAGE)都是标准 IPC handler,WS dispatcher 透明转发(阶段1.5 zero-regression)
- `ipc.ts` L1635:`filePanelService.on('filePanelUpdated', ...)` → 广播,daemon 模式走 ClientRegistry(阶段1.2)

**port 不冲突**:file-panel HTTP port(settings.filePanel.port,127.0.0.1)与 WS daemon port(12580)是两个独立服务。listenWithFallback 兔回退。

### 1.2 SSH 模块远程 ✅

```
client A:createSession(kind:'ssh', sshProfileId, path)
  │ WS command
  ▼
daemon B session-manager.createSession(L602 isSsh = pathRef.kind === 'ssh')
  │ node-pty 在 daemon B 机器 spawn:ssh <第三台 C>
  │ ssh 进程跑在 B,连 C
  ▼ PTY 字节流
经 WS 推 client A → xterm 显示
```

**代码审查证据**:
- `session-manager.ts` L602:SSH spawn 完全在 main 进程(daemon B),用 node-pty spawn `ssh`。不依赖 client 机器
- daemon 模式下 sessionManager 在 daemon 分支前初始化,createSession handler 注册在 dispatcher
- 字节流经 WS event 推 client(阶段1.3 Transport-Ws)

**含义**:用户在 client A 上操作,能通过 daemon B 中转 ssh 到第三台 C。这正是"远程后端让 Marina 成为多跳跳板"的价值。

### 1.3 tmux / ControlMaster ✅

tmux 持久化附着在 ssh 进程所在机器(daemon B)。daemon B 的 ssh 连 C 时,tmux server 跑在 B(或 C,看 ssh 配置)。ControlMaster socket 也在 B。

**代码审查证据**:`session-manager.ts` SSH 分支的 tmux 逻辑(forceTmuxChoice / ControlMaster)都在 daemon main 进程执行,持久化文件/socket 在 daemon B 本地。client A 断开/重连不影响(daemon B 的 ssh + tmux 继续)。

这正是阶段3"断网不丢 session"在 SSH 场景的延伸:client A 断线 → daemon B 的 ssh/tmux 还在 → 重连后接管。

## 2. 为什么免费工作(方案论点验证)

方案 §I.2 / §IX 核心论点:**Marina 已是前后端分离(Main / Renderer / IPC),远程后端不新拆逻辑,只把传输层接缝从 Electron IPC 拉长成 WS**。本次验证确认:

| 层 | 远程后端改动 | 验证 |
|----|------------|------|
| Renderer(含 FilePanel / MarkdownViewer) | 整体复用,一行不改 | ✅ client A 的 FilePanel 组件原样工作,数据源换成 WS |
| Main 业务(SessionManager / PathManager / FilePanelService / SSH / Settings) | 整体复用 | ✅ daemon B 跑全套 manager,file-panel HTTP + SSH spawn 都在 daemon |
| 传输层 | Electron IPC → WS | ✅ 阶段1 Transport-Ws,dispatcher 透明转发 |
| 认证 | 新增 token + safeStorage | ✅ 阶段2 |
| 重连 | 新增 backoff + resumeClientId | ✅ 阶段3 |

**file-panel 免费的关键**(方案 §I.2):agent 注入的 `MARINA_SERVICE=127.0.0.1` 指向**本地**。本地模式下指 client 自己;远程后端模式下指 **daemon 自己**(agent 跑在 daemon 机器)。两种情况都是"agent 与 Marina 后端同机",file-panel 的本机 fs.readFile/fs.watch 机制原样工作。SSH 方案之所以架构性失效(R6),正是因为 agent 与 Marina 后端不同机;远程后端修复了这个前提。

## 4. 用户测试指南(检查点 M1:A 通过 B 的 Marina 完整工作流)

### 准备
- 两台机器 A(client)+ B(daemon),WireGuard 直连(或同 LAN)
- B:`marina --headless --daemon`,托盘复制 token
- A:设置 → 远程后端 → 加 profile(B)→ "在新窗口打开"

### 测试 1:file-panel 远程(预计 2 分钟)
1. A 远程窗口在 B 上开一个 session(双击 B 的某收藏路径)
2. session 里跑一个 agent(或脚本),让它 POST 打开文件:
   ```bash
   curl -X POST "$MARINA_SERVICE/open-file" \
     -H "Authorization: Bearer $MARINA_TOKEN" \
     -H "Content-Type: application/json" \
     -d "{\"terminal\":\"$TERMINAL_ID\",\"path\":\"/etc/hostname\"}"
   ```
3. **预期**:A 的窗口右侧 file-panel 出现 hostname 文件,内容是 **B 的 hostname**(不是 A 的)
4. 点 panel 里别的文件 → A UI 切换 → B 读取
5. **失败现象**:panel 不出现 / 内容是 A 的 hostname(说明 env 注入或 HTTP 路由错)

### 测试 2:SSH 三跳(预计 5 分钟)
1. B 上配 SSH profile(连第三台 C)
2. A 远程窗口:侧栏应显示 B 的 SSH path(如果 B 有收藏 C 的 SSH path)
3. 双击该 SSH path → session 建立 → A 的 xterm 显示 **C 的 shell**(经 B 中转)
4. 在 C 的 shell 里跑命令,确认是 C 的环境(`hostname` 应是 C)
5. **预期**:A 操作 → B 中转 ssh → C 执行,字节流回 A

### 测试 3:tmux 持久化跨重连(预计 3 分钟,依赖阶段3)
1. A 远程窗口开一个 SSH session(到 C),在 C 里 `tmux new -s work`,跑 `top`
2. **断 A 的网络**(不是 B)
3. B 上 ssh + tmux 继续(top 在 C 的 tmux 里跑)
4. 恢复 A 网络 → RemoteTransport 重连 → reload
5. session 还在(owner=null),点进去 → claim → `tmux attach -t work` → 看到 top 还在跑
6. **预期**:断网期间 top 没停(C 的 tmux),重连后接管看到完整输出

## 5. 已知远程 UX 限制(reviewer 补充)

以下不是 bug,是“Renderer 整体复用”论点的边界——核心链路(file-panel 文件预览/SSH/tmux)免费工作,但部分系统命令在远程下 UX 不符 client 预期(符合方案 §VI.5“数据归 daemon B”):

- **MarkdownViewer 超链接**:点 markdown 里的链接 → `SYSTEM_OPEN_EXTERNAL`(ipc.ts L1136 `shell.openExternal`)在 **daemon** 执行 → daemon 机器弹浏览器,client 无反应。
- **“在 Explorer 中显示”**(session 右键 / Sidebar):`SYSTEM_SHOW_IN_EXPLORER` 在 daemon 开文件管理器,client 看不到。
- **设置页“打开数据/日志目录”**:`SYSTEM_OPEN_DATA_DIR`/`OPEN_LOGS_DIR` 在 daemon 开。打开 B 的数据目录**符合设计**(数据归 B),但 client 用户预期可能是本地。
- **dialog 类命令**(BOOKMARK_PICK_FOLDER 等):WS 下 `sender=undefined`,远程不支持(ipc.ts L267-269 已明示设计)。

这些在 V1 可接受(用户明示是远程后端),后续可加“远程窗口标题栏标记 + 这类命令弹提示”。**不影响 file-panel/SSH/tmux 三链路核心。**

**ws maxPayload**:`transport-ws.ts` 未显式设,走 `ws` 库默认(100MiB)。file-panel image 10MB base64 ≈13MB 远小于默认。M1 验收建议压一下大图确认。
