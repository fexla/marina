# diff 高亮 —— 解决方案

> **状态**:📋 方案待开发者裁决后实施
> **起因**:开发者否决了 v0.3.0 CP-Git 自测报告里"diff 高亮留 v0.3.1 (YAGNI)"的决定,要求必须高亮。
> **结论预告**:✅ 行级高亮零依赖即可做(~70 行),**不需要装任何新包**。词级 (intra-line) 与并排 (side-by-side) 才是需要决策的增强项。

---

## 0. 问题复盘:为什么 v0.3.0 落地版没有高亮

当前实现(已提交 commit `a801b5c`):

- `detectFileKind` 把 `.diff`/`.patch`/`.rej` 加到 `TEXT_EXT`,归 `'text'`
- FileViewer 按 `kind` 分发 → 走既有 `TextViewer`
- TextViewer 是单 `<pre>` + 整段文本节点,**零行结构**

**根因**:我当时把"diff 高亮"想成"自写词法解析器"的活,用 YAGNI 推迟。这是错的:

- **行级高亮**(add 行绿底 / del 行红底 / hunk 蓝 / file header 粗体)只需要 `line.charAt(0)` 分类,**40 行组件 + 30 行 CSS**,零依赖。GitHub 默认视图就是这个。
- **词级高亮**(行内只标改动的词,LCS 算法)才是复杂活,~150 行,**但 GitHub 默认也不开**(需点按钮)。
- 我把两者混为一谈,导致把基础体验当增强砍了。

本方案把**行级高亮作为 v0.3.0 必交付项**重新立项,词级/并排作为明确分开的增强项评估。

---

## 1. 现状事实(已核实)

| 项 | 现状 |
|---|---|
| 现成高亮库 | ❌ 无(package.json 无 highlight.js / shiki / prism / monaco / diff2html / react-diff-viewer) |
| TextViewer 结构 | 单 `<pre className="file-text-viewer">{content.text}</pre>`,整段文本节点 |

| diff 字体/换行 | `white-space: pre-wrap; word-break: break-all; font-size: 12px; line-height: 1.5`(TextViewer 既有) |

## 2. 三个方案对比

### 方案 A:行级 CSS 高亮(零依赖,推荐)★

**做法**:
1. 新增 `FileKind = 'diff'`(types.ts 枚举加一项)
2. `detectFileKind`:`.diff`/`.patch`/`.rej` → 返回 `'diff'`(从 TEXT_EXT 挪出)
3. 新组件 `DiffViewer.tsx`:按行 split,`charAt(0)` 分类,每行包一个 `<div class="diff-line diff-xxx">`
4. `FileViewer` 加 `case 'diff': return <DiffViewer/>`
5. CSS:6 个类(add/del/hunk/file-header/meta/ctx),用现有主题色变量

**代码量**:`DiffViewer.tsx` ~50 行 + CSS ~35 行 + 3 处小改动(types/detectFileKind/FileViewer)

**视觉效果**:GitHub 风格的行级着色
```
diff --git a/x.ts b/x.ts        ← file-header(粗体,info 色)
index abc..def 100644           ← meta(淡灰)
@@ -10,3 +10,4 @@               ← hunk(蓝/info 色,粗体)
 unchanged line                 ← ctx(默认色)
-removed line                   ← del(红底 / 红字)
+added line                     ← add(绿底 / 绿字)
```

**性能**:
- 1000 行:~10ms render,无感
- 1 万行:~80ms,可接受
- 5 万行(2MB 上限附近):~400ms 首屏 jank,可察觉但不致命
- **真实场景**:PR diff / 工作区 diff 99% < 500 行,完全无感

**依赖边界**:✅ 零新依赖,无需开发者批准(AGENTS.md §1.2 边界 2 不触发)

**不做**(本方案明确排除):
- 词级高亮(intra-line word diff)
- 并排视图(side-by-side)
- 行号 gutter
- 折叠未变行
- 语法着色(把 `+const x = 1` 里的 `const` 染成关键字色)

**评估**:**这就是 v0.3.0 应该交付的版本**。GitHub / GitLab / VS Code 的默认 diff 视图本质就是行级 + 前缀字符,已经足够可读。

---

### 方案 B:引入 highlight.js(需装包)

**做法**:`npm i highlight.js`,在 DiffViewer 里 `hljs.highlight(text, { language: 'diff' }).value` 出 HTML,`dangerouslySetInnerHTML`。

**优点**:免费拿到 diff 语法 + 未来可给代码文件也加高亮(README / .ts / .py 都能用同一库)。

**代价**:
- **包体积**:highlight.js 全量 ~600KB,即使只 import diff 语言也 ~50-80KB gzipped(它有运行时注册机制,tree-shaking 效果一般)
- **依赖边界**:违反 AGENTS.md §1.2 边界 2"能不装就不装"的精神,需开发者批准
- **CSP**:highlight.js 输出 HTML 需 `dangerouslySetInnerHTML` —— CSP 当前对 inline HTML 不友好(需要审 unsafe-inline,或信任其输出做 sanitize,增加复杂度)
- **过度匹配**:我们只需要 6 种行级类,highlight.js 是为几百种语言的 token 着色设计的,大材小用

**评估**:**不推荐**。用一个 600KB 库解决 40 行代码的事,且碰 CSP / 依赖边界。若以后真要给代码文件加语法高亮,那时再单独评估(且那时仍可先做方案 A 的 diff)。

---

### 方案 C:引入 react-diff-viewer-continued(需装包)

**做法**:`npm i react-diff-viewer-continued`,直接用其 `<ReactDiffViewer oldValue={} newValue={} splitView={} />`。

**优点**:开箱即用,并排 + 行号 + 词级高亮全有,视觉最接近 VS Code。

**代价**:
- **包体积**:~150KB gzipped + 拉入 `diff` (jsdiff) 依赖 ~50KB
- **依赖边界**:同样触发 §1.2 边界 2
- **输入格式不匹配**:它要 `oldValue`/`newValue`(两段原始文本),我们的输入是 **unified diff 文本**(git 输出)。要么:
  - (a) 改 GitService 不再产 unified diff,而是分别读 HEAD 版本 + 工作区版本传给 viewer —— 改后端,复杂
  - (b) 用 `diff` (jsdiff) 库把 unified diff parse 回结构化对象再喂 viewer —— 多一层 parse,且 jsdiff 的 `parsePatch` 不完美
- **产品哲学**:并排视图是 IDE 级能力,我们刻意不做 IDE(§13.2);用了会推动用户期望往 IDE 滑

**评估**:**不推荐**。输入格式适配成本高,且滑向 Git GUI/IDE 边界。

---

## 3. 推荐决策

**v0.3.0 交付 = 方案 A**(行级 CSS 高亮,零依赖)。

理由:
1. **零依赖**,不碰 §1.2 边界 2,无需批准即可实施
2. **代码量小**(~85 行,与 commit `a801b5c` 把 `.diff` 归 text 的改动量同级)
3. **视觉达标**:GitHub 默认 diff 就是行级,用户认知里"diff 高亮"默认指这个
4. **不滑边界**:不做并排 / 词级 / 行号 / 语法着色,保持"只读浏览器"定位
5. **可演进**:未来要增强,DiffViewer 是独立组件,加词级高亮只改这一个文件

**词级高亮 / 并排视图**:明确记为 v0.3.1+ 候选,**不进 v0.3.0**。若用户反馈行级不够,那时再评估方案 C(并排)或自写 LCS(词级)。先上线拿到反馈,再决定增强方向。

## 4. 方案 A 实施细节

### 4.1 类型层(`src/shared/types.ts`)

```typescript
export type FileKind = 'text' | 'markdown' | 'image' | 'diff' | 'unknown';
//                                            ^^^^^ 新增
```

**影响面**:`exactOptionalPropertyTypes` 下 `OpenedFile.kind` 加值是安全的;FileViewer 的 `switch` 有 `default` 兜底,加 case 前未知 kind 走 UnknownView(不会崩)。

### 4.2 检测层(`src/shared/file-kind.ts`)

```typescript
// 把 .diff/.patch/.rej 从 TEXT_EXT 挪到独立判定
const DIFF_EXT = new Set(['diff', 'patch', 'rej']);

export function detectFileKind(fileName: string): FileKind {
  const lower = fileName.toLowerCase();
  if (TEXT_NO_EXT.has(lower)) return 'text';
  const ext = extOf(lower);
  if (MARKDOWN_EXT.has(ext)) return 'markdown';
  if (IMAGE_EXT.has(ext)) return 'image';
  if (DIFF_EXT.has(ext)) return 'diff';   // ← 新增,在 text 之前判
  if (TEXT_EXT.has(ext)) return 'text';
  return 'unknown';
}
```

### 4.3 后端(`src/main/file-panel-service.ts`)

`readFile` 的返回类型 `ReadFileResponse` 当前按 `file.kind` 分:text/markdown → 返 `{kind, text, truncated}`,image → dataUrl,unknown → message。**需加 `'diff'` 分支**,与 text 同处理(读字符串 + 截断):

```typescript
// text / markdown / diff 同路径读
if (file.kind === 'text' || file.kind === 'markdown' || file.kind === 'diff') {
  const buf = await fs.readFile(abs);
  const truncated = buf.byteLength > MAX_READ_TEXT_BYTES;
  const text = truncated ? buf.subarray(0, MAX_READ_TEXT_BYTES).toString('utf8') : buf.toString('utf8');
  return { kind: file.kind, text, truncated };
}
```

**protocol.ts** 的 `ReadFileResponse` 类型相应加 `'diff'`。

### 4.5 新组件(`src/renderer/components/file-panel/DiffViewer.tsx`)

```tsx
/**
 * @purpose 渲染 unified diff,行级着色(GitHub 风格)。零依赖:只按行首字符分类。
 *         不做词级 / 并排 / 行号 / 语法着色(见 docs/方案-diff高亮.md §3)。
 */
import { useMemo } from 'react';
import type { OpenedFile } from '@shared/types';
import { useFileContent } from './useFileContent';
import { useTranslation } from '../LanguageProvider';

interface ViewerProps { sessionId: string; file: OpenedFile; }

type DiffLineKind = 'file-header' | 'meta' | 'hunk' | 'add' | 'del' | 'ctx' | 'no-newline';

function classify(line: string): DiffLineKind {
  if (line.startsWith('diff --git') || line.startsWith('---') || line.startsWith('+++'))
    return 'file-header';
  if (line.startsWith('@@')) return 'hunk';
  if (line.startsWith('index ') || line.startsWith('similarity ') || line.startsWith('rename ') || line.startsWith('copy ') || line.startsWith('new file') || line.startsWith('deleted file') || line.startsWith('old mode') || line.startsWith('new mode') || line.startsWith('Binary files'))
    return 'meta';
  if (line.startsWith('\\ No newline at end of file')) return 'no-newline';
  if (line.startsWith('+')) return 'add';
  if (line.startsWith('-')) return 'del';
  return 'ctx';
}

export function DiffViewer({ sessionId, file }: ViewerProps): JSX.Element {
  const { tx } = useTranslation();
  const content = useFileContent(sessionId, file.path, file.mtimeMs);

  // 按行分类只在 content 变化时算一次;500-2000 行的 diff 分类 ~1ms。
  const lines = useMemo(() => {
    if (!content || content.kind !== 'diff') return null;
    return content.text.split('\n').map((line, i) => ({
      key: i,
      kind: classify(line),
      text: line,
    }));
  }, [content]);

  if (!content) {
    return <div className="file-viewer-loading">{tx('加载中…', 'Loading…')}</div>;
  }
  if (content.kind !== 'diff') {
    return <div className="file-viewer-error">{tx('内容类型不匹配', 'content kind mismatch')}</div>;
  }
  if (!lines) return null;

  return (
    <pre className="diff-viewer">
      {lines.map((l) => (
        <div key={l.key} className={`diff-line diff-${l.kind}`}>
          <span className="diff-line-sign">{signFor(l.kind)}</span>
          <span className="diff-line-text">{l.text.replace(/^[+\-\\ ]/, '')}</span>
        </div>
      ))}
      {content.truncated && (
        <span className="file-truncated-mark">
          {'\n'}
          {tx('…(diff 过大,仅显示前 2MB)', '…(diff too large, showing first 2MB only)')}
        </span>
      )}
    </pre>
  );
}

function signFor(kind: DiffLineKind): string {
  switch (kind) {
    case 'add': return '+';
    case 'del': return '-';
    default: return ' ';
  }
}
```

**关键设计点**:
- **`useMemo` 分类**:content 变化才重算,切 tab / 重渲染不重复分类
- **`diff-line-sign` 独立槽**:把 `+`/`-` 符号放单独 span,文本 span 去掉前缀字符。这样:(a) 复制行内容不带 `+`/`-` 更干净;(b) 符号可单独弱化颜色(避免与行底色重复刺眼)
- **`pre-wrap` 保留**:超长行 wrap,与 TextViewer 一致
- **key 用 index**:diff 行没有稳定 id;但 content 变 = 整个列表重建,index 在单次渲染内唯一即可(不重排)

## 5. 性能与边界

### 5.1 性能

| diff 行数 | render 耗时(估) | 体验 |
|---|---|---|
| < 500(典型) | < 5ms | 无感 |
| 1k-5k(大改) | 10-50ms | 无感 |
| 1 万+(巨型 diff) | ~80ms | 可接受 |
| 5 万(2MB 上限) | ~400ms | 首屏 jank 可察觉 |

**已有保护**:main 端 `MAX_DIFF_TEXT_BYTES = 2MB` 截断(对齐 file-panel text 上限)。真实场景 99% < 500 行。

**若未来用户报 jank**(>1 万行):加简单"前 N 行渲染 + 展开更多"按钮,或引入 `react-window` 虚拟化(那时才装包)。**v0.3.0 不做**,先上线拿反馈。

### 5.2 边界(不滑向 Git GUI)

本方案**明确不做**:
- ❌ 词级高亮(intra-line word diff)—— GitHub 默认也不开
- ❌ 并排视图(side-by-side)—— IDE 级能力,滑向 §13.2
- ❌ 行号 gutter —— 增加列宽复杂度,边际收益低
- ❌ 折叠未变上下文 —— 增加交互,违反"用户决策最少化"
- ❌ 代码语法着色(把 `+const` 的 `const` 染关键字色)—— 滑向 IDE,且需引入语法库

这些是**有意识的克制**,对齐 §13.2 / §14.6 / ADR-017 的"只读浏览器,不做 Git GUI"定位。diff 高亮到"行级"为止。

### 5.3 依赖边界

✅ **零新依赖**。方案 A 不碰 AGENTS.md §1.2 边界 2,无需开发者批准即可实施。
