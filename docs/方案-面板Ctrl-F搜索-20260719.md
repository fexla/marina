# 方案：面板 Ctrl+F 搜索（VS Code 风格）

**日期**：2026-07-19
**版本目标**：v0.3.1（增强回合后续）
**状态**：⏳ 等待开发者确认范围与决策点

---

## 1. 需求复述

> 「每个面板能够按 Ctrl+F 弹出一个搜索框，就像 VS Code 里面那样。输入东西，它可以帮我跳转到对应的位置。」

VS Code 的 Ctrl+F 在不同区域语义不同：
- **编辑器区域**：当前文件内查找，跳转到匹配行/位置（高亮所有匹配，Enter 下一个）
- **侧栏 Explorer**：不弹 Ctrl+F 框（它用 Ctrl+P Quick Open），但 Explorer 顶部有 filter 输入框，输入即过滤文件树
- **Search 面板**：Ctrl+Shift+F 全局跨文件搜索（独立面板）

用户要的是「Ctrl+F 弹框 + 输入跳转」的统一体验。Marina 当前面板布局：
- 右 dock：Files（文件树）/ Git / Opened（已打开文件列表）
- 主区域：Terminal（已有 Ctrl+F）/ FileViewer（代码/文本/markdown/diff 查看）

---

## 2. 现状调研

| 区域 | 现状 | 搜索语义候选 |
|---|---|---|
| **Terminal** | ✅ 已有 Ctrl+F（xterm SearchAddon + 搜索栏：输入/命中数 x/N/上/下/大小写/关闭） | 不动 |
| **Files 文件树** | ❌ 无 | 过滤文件名（输入即收窄，匹配高亮，祖先目录保留） |
| **Git 面板** | ❌ 无 | 过滤变更文件路径 |
| **Opened 已打开列表** | ❌ 无 | 过滤已打开文件名 |
| **FileViewer（text）** | ❌ 无，`TextViewer` 是整块 `<pre>` 无行级 DOM | 文件内查找 + 跳转行 |
| **FileViewer（markdown）** | ❌ 无，`MarkdownViewer` 渲染 HTML | 文件内查找（在渲染后的文本节点搜，或回退源码搜） |
| **FileViewer（diff）** | ❌ 无，`DiffViewer` 已是行级 DOM（v0.3.0 双层高亮） | diff 行内查找 + 跳转行 |

**可复用资产**：
- 终端搜索栏的 UI 样式（`.terminal-search-bar` / `-input` / `-count` / `-btn`）+ 交互（Enter 下一个 / Shift+Enter 上一个 / Esc 关闭 / Aa 大小写）→ 抽成共享 `<SearchBar>` 组件
- `FileListRow`（已统一列表渲染）→ 列表过滤后复用渲染
- LayoutHost 的 PanelStack 已知 active panel → 可在 dock 层路由 Ctrl+F

## 3. 核心设计

### 3.1 快捷键路由：dock 级 window keydown

Terminal 的 Ctrl+F 走 xterm 内部 `attachCustomKeyEventHandler`（xterm 吃掉键，不冒泡到 window）。
**其他面板没有 xterm**，需要一个 window 级 keydown 监听：

```
window keydown (capture phase):
  if Ctrl+F 且 焦点不在 terminal/input:
    找到当前 session 的 active dock panel
    打开该 panel 的搜索框
    preventDefault
```

放在 LayoutHost（或一个 `usePanelSearchShortcut` hook），因为只有它知道当前 active panel。

### 3.2 共享 SearchBar 组件

抽 `src/renderer/components/common/SearchBar.tsx`：
- props：`value` / `onChange` / `onClose` / `onNext` / `onPrev` / `caseSensitive` / `matches` / `current`
- 样式复用终端搜索栏（重命名 `.search-bar` 通用类，终端也切过去用同一个组件）
- 键盘：Esc 关闭 / Enter 下一个 / Shift+Enter 上一个

### 3.3 两种搜索语义

**A. 列表过滤型（Files / Git / Opened）**
- 输入即过滤：匹配项保留，不匹配隐藏
- 匹配文字高亮（`<mark>`）
- Files：保留匹配文件的祖先目录（让用户看到路径上下文）
- 无"下一个"概念（列表是平面的），Enter 选中第一个匹配并打开/跳转
- SearchBar 的 `matches`/`current`/上/下按钮隐藏（不适用）

**B. 文件内查找型（FileViewer: text/markdown/diff）**
- TextViewer 改行级渲染（每行 `<div data-line="N">`）
- 输入后：高亮所有匹配行，`matches = 总命中数`，`current` 跟踪当前焦点
- Enter/上下：`scrollIntoView` 到下一个/上一个匹配行
- 命中数显示 `x/N`（和终端一致）

## 4. ⚠️ 需要开发者确认的决策点

### 决策 1：本轮做哪些面板？

| 选项 | 范围 | 工作量 |
|---|---|---|
| **① 最小** | Files / Git / Opened 三个列表型面板（过滤型 A） | 小（~1-2 commit） |
| **② 中等** | ① + FileViewer text/diff 文件内查找（B） | 中（TextViewer 改行级 + scrollIntoView，~3 commit） |
| **③ 完整** | ② + markdown 查找 | 中大（markdown 渲染后 DOM 搜文本节点，复杂） |

**推荐 ②**：列表过滤 + 代码/diff 内查找，覆盖 90%「跳转到位置」诉求。markdown 本质是渲染后阅读，文件内查找价值低（且实现复杂），记 BACKLOG。

### 决策 2：搜索框位置？

| 选项 | 说明 |
|---|---|
| **① 面板顶部**（每个面板组件内部） | 搜索框在面板内容上方，和面板内容一起滚动？还是 sticky？ |
| **② dock header 下方**（LayoutHost 层，dock 级共享） | 一个框服务当前 active panel，位置固定不随面板内容滚 |

**推荐 ②**：dock 级共享框，位置稳定，Ctrl+F 体验一致（像 VS Code 编辑器顶部的小搜索条）。实现上 LayoutHost 持有 `searchVisible` state，按 active panel 渲染框 + 把 query 传给 ActivePanel。

### 决策 3：列表过滤型的交互细节

- **输入即过滤** vs **Enter 才过滤**？→ 推荐**输入即过滤**（VS Code Explorer filter 风格，响应快）
- **Files 过滤时是否展开匹配的祖先目录**？→ 推荐**是**（否则过滤后看不到匹配项）
- **无匹配时**显示空态「无匹配文件」
- **清空搜索**恢复完整列表

### 决策 4：快捷键边界

- Ctrl+F 只在**右 dock 面板有焦点 / 鼠标在 dock 内**时触发？还是**全局**？
  - 全局：用户在任何地方按 Ctrl+F 都打开 dock 面板搜索（可能和终端 Ctrl+F 冲突——终端有焦点时 xterm 吃掉，不冒泡，所以不冲突）
  - 推荐**全局**（焦点在终端时 xterm 拦截走终端搜索；焦点不在终端时走面板搜索），行为可预测
- Esc 关闭搜索框（和终端一致）

### 决策 5：大小写敏感默认？

- 终端搜索默认**不敏感**
- 推荐**一致**（不敏感，Aa 按钮可切换）

## 6. 不做的事（边界）

- ❌ **全局跨文件搜索**（VS Code Ctrl+Shift+F）：那是另一个独立搜索面板，超出本轮
- ❌ **正则搜索**：终端搜索也不支持，保持一致
- ❌ **替换（Replace）**：Marina 是只读查看器，无编辑语义
- ❌ **markdown 渲染后查找**：复杂度收益比低（记 BACKLOG）
- ❌ **搜索历史**：YAGNI
