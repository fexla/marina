# Marina 的 AI Harness 设计总结

> 把 `AGENTS.md`(v1.3) 当作一份"提示词工程作品"来读,提炼它背后的设计逻辑。
> 这不是目录复述,而是回答一个问题:**这套规范为什么这么设计?**

---

## 0. 这套 harness 在解决的核心矛盾

**信任不对称 + YOLO 自治**:

- AI 被授权"端到端不打扰地构建整个产品"
- 但"这个项目没人 review 你的代码",人类只能在**事后**接手
- 于是整套设计围绕两个支点展开:
  1. **事前约束** —— 用边界、检查点、文档契约把 AI 圈在安全区
  2. **事后可追溯** —— 注释、commit 颗粒度、BLOCKED、日志,让人类能读懂、能 bisect、能接手

一句话:**让 AI 在边界内最大化自治,同时让人类随时能以最低成本接管。**

---

## 1. 文档优先级与阅读契约(看什么文档)

### 强制阅读顺序(不可跳)
1. `docs/软件定义书.md` —— 产品设计共识(必懂第 2 章哲学、第 8 章状态机)
2. `AGENTS.md` —— 本文件
3. `docs/ipc-protocol.md` —— IPC 规格(不存在就要先一起定,不许脑补)
4. `README.md`

### 三条文档纪律
- **文档缺失即停**:任何必读文件不存在或残缺 → 立刻停下通知开发者,**不许自己脑补**
- **文档即共识**:不靠记忆,靠读;设计哲学以软件定义书为准
- **文档自演化**:版本号(v1.3)+ 变更日志(top of file)+ ADR 编号 + 勘误回合沉淀,让文档能追溯每一次共识变更

### 勘误回合(checkpoint errata)机制
- 每个检查点测试后,问题整理进 `docs/cpN勘误.md`(自由文本)
- 勘误是**检查点工作流的一部分**,不是特殊状态
- 修复完要双向沉淀:软件定义书加"v{X.Y} 起 / CP-{N} 勘误"标注 + AGENTS.md 更新完成标志 + CP 自测报告加章节

## 2. 工作模式:YOLO + 三条红线(怎么做事)

### 默认行为是"继续干活",不是"等指示"
可以自主决策(命名/组织/API 设计/测试选型/调试策略)、自主执行(写码/构建/测试/commit/装限定依赖)。

### 三条永远不能越的边界(用"具体例子"而非抽象原则表达)
| 边界 | 内容 | 设计要点 |
|------|------|----------|
| **破坏性操作** | 删文件、force push、改 .git、碰用户数据目录、卸载包、改注册表 | 列具体动词:"删除/reset/clean",防止 agent 钻空子 |
| **超出技术栈** | 想加新 npm 包 → 停下问 | 理由写明:agent 历史常引入冷门/废弃/有安全风险的包。例外也写明:`@types/*`、小工具(uuid/debounce)、测试包 |
| **产品哲学** | "永远不做"列表是红线 | 每条红线配**具体反例**("加应用内快捷键""加 main window""关窗弹确认"),让 agent 没法绕 |

### 关键设计:**边界用例子表达,不用原则表达**
附录 C "你绝对不能问的事" 直接列出"答案永远是'不'"的问题,并标注**唯一例外**(如 ADR-013 的 Linux 关窗确认,触发条件写得极窄)。这避免 agent 用"虽然……但是……"找借口。

---

## 3. 检查点阀门(怎么做计划)

### 把开发切成 N 个 checkpoint,每个 CP 强制停
- 与软件定义书 Phase 路线图对齐
- 每个 CP 用独立 feature branch(`checkpoint-N`),通过前不许 merge main
- 通过后代码"**封箱**",后续不许顺手重构(除非强制要求/修 bug/明确指示)

### 每个 CP 必须三件产出
1. **自测报告**(`CP-N-self-test.md`)—— 你自己跑过什么、已知不工作、没测的东西
2. **用户测试指南**(`CP-N-user-test-guide.md`)—— "写得像给不熟悉项目的人看",含准备/步骤/预期/失败现象/日志路径
3. **明显标记** `🛏️ CHECKPOINT N: 等待开发者测试`

### 设计意图
检查点不是"麻烦事",是**方向校准阀门**。一次跑偏到 CP-4 才发现,损失 > 4 次检查点开销之和。

## 4. 沟通协议(怎么和用户沟通)

### 用 emoji 做硬信号(降低歧义)
- `🛏️ CHECKPOINT N` —— 等待测试
- `🛑 BLOCKED` —— 卡死求助
- `🎉 ALL CHECKPOINTS COMPLETE` —— 全部完成

### 三类问题分层(附录 A/B/C 是沟通成本的精算)
| 类型 | 处理 | 例子 |
|------|------|------|
| **必须问** | 破坏性操作/新依赖/哲学冲突/文档模糊 | 删文件、装新包 |
| **不必问** | 看边界/技术栈就能自答 | "能不能用 X 工具""代码风格" |
| **永远不问** | 答案永远是"不" | vim 模式、project 概念、关窗确认、force push、mock 核心模块 |

### 检查点等待期间允许做的小事
整理文档、更新 CHANGELOG、补注释、修 lint —— **不许"乘等待之机"重构旧代码**。

---

## 5. 调试熔断:10 轮规则(你可能没想到的)

### 触发条件(单一问题反复修)
- 同一测试失败 10 次以上
- 同一段代码改 10 次还在
- commit 里"修复 X"出现 10 次以上
- 怀疑过 5 个以上根因都不对

### 触发后必须做
1. **停止所有修改**(保留现场,不回退)
2. 写 `BLOCKED.md`(问题/复现/尝试过的思路/最强猜测/关键文件/诊断日志 5-20 行)
3. 输出求助信号
4. 等待,不再猜

### 设计意图
"10 轮没解决 = 你对问题的理解从一开始就错了",这时人介入比 agent 继续猜效率高 100 倍。**不鼓励撑到第 10 轮**,5 轮就走投无路就提前 BLOCKED 也合理。

### 不算"卡 10 轮"的情况(避免误触发)
不同功能各跑几次、不同 bug 各 1-2 轮、CI 重跑 —— 这些是正常工作。

## 6. 代码可维护性工程(注释哲学,你可能没想到的)

### 为什么注释要求这么高
"代码 review 是质量保证"的替代品。AI 犯错时,开发者只能事后接手;读不懂就只能扔掉重写。所以注释标准是:**没读过 AI 思考过程的人,一小时内能理解任何一个文件**。

### 必须写注释的地方(全部给了模板)
- **文件头注释** —— `@file/@purpose/@关键设计/@对应文档章节/@不要在这里做的事`
- **公开函数 + 复杂私有函数** —— 含 `@throws/@副作用/@常见问题排查`
- **状态机相关代码** —— 顶部画 mini 状态图或引用文档章节
- **"为什么这么写"而不是"这行做什么"** —— 例:`setImmediate` 而非 `nextTick`,因为 nextTick 优先级太高会饿死 PTY data 事件

### 错误信息标准化
`throw` 必须含:操作名 + 关键参数值 + 至少 2 个可能原因 + 建议下一步 + 日志路径。

### 日志分级约定
`debug`(IPC/字节流)/ `info`(session 创建销毁、窗口开关)/ `warn`(PTY 异常退出)/ `error`(致命),带 `[模块名]` 前缀 + 关键参数。"开发者排查问题时,日志是他的眼睛。"

### 不要写的注释
`// 设置 x 为 5`(重复代码)、`// TODO: fix later`(要写具体修什么)、注释掉的代码(直接删,git 有历史)。

## 7. 测试策略的取舍(你可能没想到的)

### 后端必测 / 前端不测,理由写得很硬
- **主进程是产品"大脑"**,状态机和 PTY 出错代价大,自动测试性价比高
- **UI 测试在 Electron 里成本高**(spectron/playwright 兼容性),收益低于人工测
- "人工测试对 UI 更好,因为人能感知**丑**和**反人类**,自动测试感知不到"

### 必测场景(后端)
- 状态机:path / session / 应用生命周期
- 核心管理器:SessionManager / PathManager / SettingsManager / WindowManager
- 协议:IPC 序列化、schema 验证
- 持久化:原子写、损坏恢复、版本迁移
- 解析:OSC 1337、字节流状态识别

### 测试要"会出错"
不只测 happy path,还要测:错误输入(null/undefined/超长串)、边界(0/1/N、到 30 上限)、并发(同 session 两窗口 claim)、异常(PTY 启动失败、JSON 损坏、磁盘写失败)。

### 覆盖率目标阶梯
CP-2 核心数据 >70% → CP-3 状态机 >80% → CP-4 整个 main >75%。**不追求 100%,追求关键路径有保护**。

## 8. Git 作为救命稻草(commit 纪律)

### 颗粒度动机
"出问题后 `git bisect` 是开发者**唯一**的救命稻草"——所以一个独立功能/bug 修复/重构 = 一个 commit。禁止"一天一个 commit 包所有"。

### Conventional Commits
`<type>(<scope>): <subject>`,type=`feat/fix/refactor/test/docs/chore/perf`,scope=`main/renderer/session/path/tray/settings/ipc/theme`。

### 分支策略
- `main`:只接通过检查点的代码
- `checkpoint-N`:每个 CP 一个分支
- `fix/xxx`:CP 失败的修复分支,基于 `checkpoint-N`
- 禁止在 main 上直接写;force push 任何分支都要先问

### Commit 前必须
lint 无错(警告可接受)+ test 通过 + prettier 格式化。

## 9. 产品哲学保护层

### "永远不做"列表(软件定义书 13.2 节)是红线
如果"加个 XXX 会简单很多"而 XXX 在列表里 → **不做,停下问**。

### 三个典型反例(直接抄进 AGENTS.md 附录 C)
- 加应用内快捷键切换 session(违反"鼠标优先")
- 加"主窗口"概念简化持久化(违反"窗口平等")
- 关窗加确认对话框(违反"窗口零成本开关")
  - **唯一例外**:Linux + 最后窗口 + 仍有非 exited session,弹同一个 `<LastSessionConfirm />`(ADR-013);触发条件写得极窄,非 Linux / 非最后窗口 / 全 exited 都禁止弹

### 设计意图
把哲学翻译成**可执行的判断**,而不是留给 agent 自己推导(它推导时常常会"合理化")。

---

## 10. 平台抽象与数据安全

### V1 只测 Windows,但代码必须 platform-aware
- 所有平台特定 API 走 `src/main/platform/`
- `windows.ts` 完整实现;`macos.ts`/`linux.ts` 留占位 throw `Not implemented`
- **不许在 `windows.ts` 之外写 `process.platform === 'win32'`**
- **不许"顺手"实现 macOS/Linux**——理由:"你不会在 macOS/Linux 上测,实现了等于发布未经测试的代码"

### 数据安全(数据是红线)
- **永不**删/改 `~/AppData/Roaming/Marina/` 下任何文件(测试也不行)
- 测试用 `os.tmpdir()` 临时目录或 memfs,测完清理
- 测试**不许 spawn 真实进程**(浏览器/shell/应用),只用 mock 或隔离环境
- 不许提交敏感数据;代码里用占位路径;`.env` 进 `.gitignore`

### 权限速查表(附录 A)
列清每条路径能读/能写/能删与否 —— 把"安全"从原则变成可查表。

---

## 11. 性能底线(不优化原则)

### 底线(V1 不追求极致)
冷启动 <3s、session ready <1s、PTY→xterm <50ms、设置同步 <200ms、稳态(10 session/2 窗口)<500MB。

### 关键设计:**保持简单是最好的性能策略**
- 超底线 2 倍以上才评估设计问题
- **不在没问题的地方做性能优化**(过度抽象、缓存、异步队列)
- 不"先实现着以后再优化"

## 12. 元设计洞见(我补充的)

这套 harness 真正高明的地方,不在某一条规则,而在几个**贯穿始终的设计原则**:

### ① 具体例子 > 抽象原则
- 边界用"删除/reset/clean"这种动词表达,而非"小心破坏性操作"
- 哲学用"加快捷键""加 main window""关窗确认"这种反例表达
- 例外条件写得极窄(ADR-013)
→ **目的:让 agent 没有钻空子的语义空间**

### ② 事前约束 + 事后追溯,双管齐下
- 事前:边界、检查点、阅读契约、测试必测场景
- 事后:注释模板、commit 颗粒度、BLOCKED、日志分级、错误信息标准化
→ **因为没有 review,只能靠"结构"补位**

### ③ 决策树式工作流,而非线性流程
第 11 章给出**正常路径 + 异常路径**两种:正常一路到 CP-5;异常分 A/B/C/D/E 五种停下的情况(卡死/新依赖/文档模糊/哲学冲突/顺手重构冲动)。每种都有明确动作。

### ④ 文档自演化机制(可追溯的共识变更)
- 顶置 `v1.X 变更` 日志
- ADR 编号(架构决策记录)
- 勘误回合把问题沉淀回软件定义书 + AGENTS.md + CP 自测报告三处
→ **文档不是静态规范,是活的共识**

### ⑤ 注释是写给"未来的自己"
"测试是写给未来的你"——CP-3 改 SessionManager 时,CP-2 的测试会告诉你有没有破坏旧逻辑。注释同理:开发者事后接手时,注释是他的眼睛。

### ⑥ YOLO 成功标准是 checklist(第 1.3 节)
不只给自由,也给**验收标准**:按软件定义书实现 V1 / 不违反"永远不做" / 每检查点停 / 后端测试覆盖 / 注释充分 / commit 清晰 / 能打包成 Windows 安装包。**自治的前提是可验收。**

### ⑦ 沟通成本精算(附录 A/B/C 三级)
- 附录 A:文件权限速查(能做的不用问)
- 附录 B:没必要问的事(自己能判断的不用问)
- 附录 C:绝对不能问的事(答案永远是"不")
→ **把"问"这个动作的成本压到最低,只在真正需要人决策时才发生**

### ⑧ 错误信息 / 日志 / 注释的"接手友好度"三位一体
三者都强调:**带模块名、带参数、带原因猜测、带下一步**。开发者排查问题时,这三者是统一的"诊断接口"。

## 13. 可直接借鉴到其他项目的设计模式

| 模式 | 一句话 | 适用前提 |
|------|--------|----------|
| **YOLO + 三红线** | 自由最大化,边界用例子表达 | 有清晰的产品哲学 |
| **检查点阀门** | 切 N 段,每段强制停 + 自测 + 用户指南 | 项目可切片验收 |
| **10 轮熔断 + BLOCKED.md** | 卡死不硬冲,保留现场 | AI 长时自治任务 |
| **文件头注释模板** | 每个文件自带"读我" | 任何无 review 项目 |
| **错误信息标准化** | throw 必含 操作+参数+原因+下一步 | 任何项目 |
| **测试分层(后端必测/前端不测)** | 把资源投在"大脑"上 | Electron/重后端项目 |
| **commit = bisect 单元** | 颗粒度服务于事后定位 | 任何项目 |
| **文档版本 + ADR + 勘误回合** | 共识变更可追溯 | 长期演进项目 |
| **权限速查表** | 把安全从原则变查表 | 涉及用户数据的项目 |
| **沟通三级分流(必问/不必问/永不问)** | 压低问的成本 | 任何 AI 协作项目 |
| **例外条件写极窄** | 给豁免口子但要极窄触发 | 有铁律但需个别 case 例外 |

## 附:这套 harness 的"人格"画像

读完 AGENTS.md,它塑造的 AI agent 形象是:

> 一个**默认往前冲、但在标线前会自动刹车**的工程师;
> **写代码时假设下个月有人要 debug 它**;
> **卡住时不会硬撑、会主动求助**;
> **不钻空子、不"顺手"、不自作主张扩大范围**;
> **把文档当共识、把 commit 当救命稻草、把注释当眼睛**。

这本质上是**用提示词工程模拟一个"高自治 + 高纪律"的高级工程师**——
而不是一个"听指令干活"的实习生。
