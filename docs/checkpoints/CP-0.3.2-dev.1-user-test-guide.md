# 0.3.2-dev.1 用户测试指南

**测试目标**：`0.3.2-dev.1` 这个 portable **累积包含** 0.3.1 和 0.3.2 两批改动。
一次测试就能覆盖两个版本的全部新功能与修复。

**产物**：`release/0.3.2-dev.1/Marina-Portable-0.3.2-dev.1-x64.exe`
（SHA-256 `aab2e86a...4284a4d0`；开发构建，不作为正式 GitHub Release 发布）

---

## 准备

1. 双击 `Marina-Portable-0.3.2-dev.1-x64.exe` 启动（portable 模式，数据落 exe 旁的 `Marina/` 目录，不碰已安装的正式版）。
2. 看到 Marina 窗口、托盘图标、侧栏（收藏/临时/最近）齐全即表示启动正常。
3. **全局失败排查**：`Marina\logs\main.log` 是你的眼睛。出现红字/异常/行为不符时，把最后 50 行贴回来。
4. 性能报告目录：`Marina\performance-reports\`（0.3.2 新增，见 A1）。

# 第一部分：0.3.2 新功能（本次重点）

## A1. 性能飞行记录器（自动）

**它是什么**：每次运行 Marina 都会自动在 `performance-reports/` 生成一份 JSON + Markdown 性能报告，常驻低开销采样，无需任何手动操作。

**测试步骤**：
1. 启动 Marina，打开一个终端用 ~30 秒。
2. 打开 `Marina\performance-reports\` 目录（可用 A3 的"打开报告目录"按钮，或在文件资源管理器手动进）。
3. 应看到一对文件：`run-<时间戳>-<短id>.json` 和 `.md`。

**预期**：
- ✅ 有一对 JSON + Markdown 文件，文件名带本次运行时间戳。
- ✅ 用文本编辑器/浏览器打开 `.md`，能看到：
  - 健康摘要（main event-loop stall 次数、最大 stall ms、CPU 峰值、RSS 峰值、采样数）
  - 最近一次采样（main CPU、RSS/heap、event-loop utilization、delay 分位数）
  - Electron 子进程表（Browser/Tab/GPU/Utility 进程数/CPU/Working set）
  - Active resources 列表
  - 操作热力图（IPC/Git/session/PTY 的 count/耗时/in-flight）
  - 业务计数与当前值（counter/gauge）
- ✅ 打开 `.json`，检查**不应**包含：你电脑的用户名、项目绝对路径、终端命令、终端输出内容。自动报告只允许数值和固定标签。

**失败现象**：目录为空 / `.json` 损坏 / 报告里出现了你的真实路径或终端内容 → 回归。

---

## A2. 按需 15 秒 V8 CPU Profile（显式）

**它是什么**：设置页可以显式采集一份 main 进程的 `.cpuprofile`（可用 Chrome DevTools Performance 面板加载），用于深度性能分析。

**测试步骤**：
1. 设置 → 高级 → 性能诊断区，找到「捕获 15 秒 CPU Profile…」按钮。
2. 点击 → 应弹出原生确认框，提示内容包含**函数名 / 本地源码路径隐私风险**。
3. 确认 → 按钮变「采集中（15 秒）…」，等 15 秒。
4. 完成后应弹 alert 提示生成路径，并自动打开报告目录。

**预期**：
- ✅ 采集期间按钮禁用，第二次点击（如果点了）应被拒绝"已有 capture 在跑"。
- ✅ 生成一份 `run-<id>-cpu-<时间戳>.cpuprofile`。
- ✅ 把 `.cpuprofile` 拖进 Chrome → `chrome://inspect` → Open dedicated DevTools → Performance 面板（Load profile），能看到火焰图。

**失败现象**：没弹确认 / 弹了但没隐私提示 / 采集不结束 / 没生成文件。

---

## A3. 性能报告入口（设置页）

**它是什么**：设置页显示本次运行的性能摘要，并提供两个操作按钮。

**测试步骤**：
1. 设置 → 高级 → 性能诊断区。
2. 看到：运行 id、采样数、stall 计数、最近 main CPU、RSS、报告文件名、是否在采 profile。
3. 点「立即刷新报告」→ 摘要里的采样数应立即 +1，报告目录的 `.json`/`.md` 时间戳更新。
4. 点「打开报告目录」→ 资源管理器打开 `Marina\performance-reports\`。

**预期**：两个按钮都能成功；摘要数字随运行时长增长。

**失败现象**：「打开报告目录」点了没反应（静默）→ 这是修复点之一，应弹错误。

---

## A4. 需求感知 Git 扫描（HOT / WARM / COLD）⭐ 核心修复

**它是什么**：Git status 不再固定 3 秒无脑扫，而是按"用户到底在不在看 Git 面板"动态调频。这是解决长期运行卡顿的**根因性**改动。

**准备**：在一个真实的 Git 仓库目录里开终端（例如 `D:\projects\xxx`）。

**测试矩阵**（用 A3 的"立即刷新报告"看 `background.hotTasks` / `background.warmTasks` gauge）：

| # | 操作 | 预期 gauge |
|---|---|---|
| 1 | 切到 Git 面板，窗口保持前台 | `hotTasks = 1`，约每 3 秒刷新一次 |
| 2 | 切到「文件」或「已打开」面板（仍选中该终端） | `warmTasks = 1`，`hotTasks = 0`，每 60 秒才扫 |
| 3 | 折叠右 dock | 同上 WARM |
| 4 | Marina 窗口失焦（点别的窗口/进游戏） | WARM |
| 5 | 切到另一个终端 session | 旧 session 的 task = NONE（停止） |
| 6 | 关闭全部窗口进托盘 | 全部 NONE |
| 7 | 重新开窗、切回该终端、切回 Git 面板 | 恢复 HOT，且**立即刷新**（不等 60 秒） |

**行为验证**（不用看 gauge，直接看 Git 面板内容新鲜度）：
- 在外部改一个文件（`echo x >> a.txt`），切到 Git 面板 → 应在约 3 秒内看到新文件出现。
- 切到「文件」面板，再外部改一个文件 → 切回 Git 面板，应**秒显缓存的旧列表**（WARM 期间保持温热），随后立即补真值刷新。

**多窗口**：开两个 Marina 窗口，窗口 A 显示终端 1 的 Git 面板（HOT），窗口 B 切到别的 → 终端 1 的 task 取最高等级 = HOT；两个不同 session 同时到期时全局最多一个 `git.exe` 在跑。

**失败现象**：切到 Git 面板后等了 60 秒才更新（没切回 HOT）/ 关窗后任务管理器仍持续看到 git.exe 周期出现 / 切回面板要等很久（没立即刷新）。

# 第二部分：0.3.1 功能（回归确认）

## B1. 长期运行不再因 Git 轮询累积拖慢系统

**背景**：0.3.1 的核心修复——退出的 session 不再每 3 秒 spawn git.exe。0.3.2 进一步把它升级为按需求调频（A4）。

**测试步骤**：
1. 在 Git 仓库开 3 个终端 session（session 1/2/3）。
2. 在每个 session 里执行 `exit`（PTY 退出，但 tab/scrollback 保留）。
3. 关闭全部窗口，进托盘。
4. 打开任务管理器，等 1-2 分钟，观察是否有 `git.exe` 周期出现。

**预期**：退出 session 后不再后台 spawn git。任务管理器里**不应**看到周期性的 git.exe。

**失败现象**：历史 session 的 git.exe 仍周期出现 → 即"长期运行越来越卡"的回归。

---

## B2. Git 面板不抢 `.git/index.lock` 且不误报"干净"

**测试步骤（锁冲突）**：
1. 在 Git 仓库开终端，确保 Git 面板在跑。
2. 在终端里手动跑 `git add .` / `git commit -m "test"`（或外部脚本高频 commit）。
3. 观察 add/commit 是否报 `Unable to create '.git/index.lock': File exists`。

**预期**：不应再撞锁（此前概率 ~15-20%）。

**测试步骤（不误报干净）**：
1. 找一个有**大量改动**的真实仓库（如 Unity 项目，几千文件变更）。
2. 切到 Git 面板，应正确列出所有改动，而不是显示"工作区干净"。

**失败现象**：commit 撞锁 / 大改动仓库被误报干净（这是 0.3.1-dev.1 的 `--no-optional-locks` 参数错放位置 bug 的回归方向）。

---

## B3. 文件条目图标按后缀细分

**测试步骤**：打开文件树，观察不同类型文件的图标：
- `.ts`/`.js`/`.py` → 代码图标
- `.json`/`.yaml` → 配置图标
- `.md`/`.txt` → 文档图标
- `.png`/`.jpg` → 图片图标
- `.zip` → 压缩图标
- `.prefab`/`.asset`/`.meta`（Unity）→ 资产图标
- `.lock` → 锁文件图标
- 目录 → folder 图标

**预期**：不再是清一色通用 file 图标，按后缀有区分。Git 面板、文件面板、文件树三处一致。

---

## B4. 文件面板顶部「当前目录 / 临时工作区」切换

**测试步骤**：
1. 开一个本地终端，切到「文件」面板。
2. 顶部应有双按钮（或单按钮，取决于有几个根）：当前目录 / 临时工作区。
3. 点切换，内容区只渲染当前选中根。
4. 选中的根**跨重启记忆**（关 Marina 重开，应记住上次选的根）。

**预期**：双根切换正常，跨重启保持选中。

---

## B5. 面板状态 / 展开记忆

**测试步骤**：
1. 文件树展开几个目录 → 切到「已打开」→ 切回「文件」→ **展开状态应保留**。
2. Git 面板树形/平铺视图切换 → 切走再切回 → **viewMode 保留**。
3. Git 面板右键菜单 / 文件树右键菜单形态一致（能力驱动统一菜单）。

**预期**：工作态（展开目录）切面板不丢；视图模式跨重启保留。

---

## B6. 树形缩进统一

**测试步骤**：文件树、Git 树、文件面板的嵌套目录，缩进视觉一致（同一深度对齐）。
此前三处各自硬编码 14px / CSS 层叠，现在统一。

---

## B7. `show-in-marina` skill 的 `marina workspace`

**仅当装了内置 skill 时测**：在某收藏项目装了 `show-in-marina` skill 的 agent（Pi/Claude Code/Codex）里。
1. 让 agent 跑 `marina workspace` → 应输出当前 terminal 的受管临时目录绝对路径。
2. agent 用这个具体路径写文件（不是 `$MARINA_WORKSPACE` 字面量）→ 文件应出现在 Marina 的「已打开」面板。

**预期**：不再在项目下错误创建名为 `$MARINA_WORKSPACE` 的字面目录。

---

## B8. 代码查看器：行号栏分离 + diff 行号

**测试步骤**：
1. 在 Git 面板点一个改动的文件 → 跳「已打开」看 diff。
2. diff 应有**行号槽**（从 hunk header 解析），ctx/加/删各自正确行号。
3. 横向滚动 diff → 行号**钉在左侧不动**（sticky），不被代码盖住。
4. 打开一个代码文件本身（非 diff）→ 同样有行号 + 语法高亮，横向滚动行号不跟着移。

**预期**：长行不换行（出水平滚动条），行号 sticky，底色跟随行高亮（如搜索当前命中行）。

---

## B9. 中键拖动平移（手型工具）

**测试步骤**：在 diff / 代码 / markdown 三个查看器里，按住鼠标中键拖动 → 应平移滚动（上下/左右）。
**预期**：不触发 Windows 原生的圆形滚动光标。

---

## B10. 后台终端的面板状态持续更新

**测试步骤**：
1. 窗口 A 持有终端 1，切到 Git 面板。
2. 开窗口 B，切到终端 2（终端 1 变 orphan，`ownerWindowId=null`）。
3. 在窗口 B 期间，终端 1 所在仓库被外部改了文件。
4. 切回窗口 A 的终端 1。

**预期**：切回时看到的是**最新**的文件列表/diff，不是旧的。（此前 orphan 期间事件被丢弃，切回看旧值。）

---

## B11. diff/代码语法高亮语言（18 种）

**测试步骤**：打开以下后缀文件的 diff，应有内层语法高亮（不是只有行底色）：
`ts/js/py/json/bash/yaml/markdown/xml/c/cpp/cs/go/rust/java/kotlin/ruby/php/sql`

**预期**：go/rust/java 等（0.3.1 新补的 7 种）有高亮；其余回退纯 diff 语言。
