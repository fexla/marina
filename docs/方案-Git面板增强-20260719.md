## 问题 2:滚动溢出(最易,先修)

### 根因
`global.css:2026` 的 `.git-panel` 只有 `padding: 4px 0`,**没有 `overflow-y: auto` 也没有高度约束**。父容器(LayoutHost 的 ActivePanel 渲染区)若是 flex 列布局,`.git-panel` 要承接剩余高度并内部滚动。

### 方案
```css
.git-panel {
  padding: 4px 0;
  font-size: 12px;
  flex: 1 1 auto;        /* 新增:承接父容器剩余高度 */
  min-height: 0;          /* 新增:flex 子项允许收缩,overflow 才生效 */
  overflow-y: auto;       /* 新增:内容超出滚动 */
  overflow-x: hidden;     /* 新增:横向不抖(路径长时 wrap 即可) */
}
```
同时确认 LayoutHost 的 ActivePanel 外层容器是 `display: flex; flex-direction: column` 且自身有确定高度(右 dock 高度由父 grid/flex 决定)。若外层不是 flex 列,需要加一层 `<div className="git-panel-scroll">` 包裹。

### 验证
造 100 个变更文件,确认出现滚动条且不撑爆 dock 高度。

### 工作量
~5 分钟(CSS)+ 验证。**独立 commit**,不阻塞其他问题。

## 问题 4:切换延迟(最影响体验,且要写进约束)

### 根因(三因素叠加)

1. **LayoutHost lazy mount**:`LayoutHost.tsx:239` `<ActivePanel key={`${activePanelId}-${session.id}`}>` —— 切走 Git tab = 卸载组件,切回 = 重新 mount。
2. **GitPanel 每次 mount 都拉**:`useEffect(() => { void loadStatus(); }, [loadStatus])`,无缓存。
3. **spawn git 二进制慢**:Windows 上 `git status` 冷启 ~100-300ms(ConGit 进程 + 索引加载),加 IPC 往返,用户感知 200-500ms。

GitService 注释(`git-service.ts:19`)明确"不缓存 status 真值,每次拉取都走 IPC,GitService 是唯一状态源" —— 这是当初 PRD §5.1.4 的"最小可行"妥协,**现在要打破它**。

### 方案(三层叠加)

**层 1:renderer 端组件级缓存(立竿见影)**

GitPanel 用 `useState` 存 snapshot,组件卸载就丢。改成**模块级 Map<sessionId, snapshot>**(或 React Context),切走保留,切回秒显。

```tsx
// 新建 src/renderer/state/git-status-cache.ts
const cache = new Map<string, { snapshot: GitStatusSnapshot; at: number }>();
export function getCached(sessionId: string): GitStatusSnapshot | undefined { ... }
export function setCached(sessionId: string, s: GitStatusSnapshot) { ... }
```

GitPanel mount 时:`const cached = getCached(sessionId); setState(cached ? {snapshot: cached} : {loading: true})` —— **有缓存先秒显**,后台再静默刷新(`loadStatus()` 照调,但 loading 态不覆盖已有 snapshot)。

**层 2:预取(session 创建/cwd 变更时就拉,不等用户点 tab)**

main 端 `applyGitAvailability`(cwd 进仓库时)已经知道"这个 session 现在是 git 仓库了",顺便调一次 `getStatus` 推给 renderer(`evt:git:status-updated`,这个事件已存在但当前只在 openDiff 后触发)。renderer 收到就更新缓存。

效果:用户点 Git tab 时,**缓存已有**,零延迟。后台文件变更通过 watcher(见层 3)持续更新。

**层 3:watcher(仓库变更主动推)**

GitService 启用 `fs.watch(repoRoot, { recursive: true })`,变更时 debounce 500ms 重算 status → emit `evt:git:status-updated`。这消除 PRD §5.1.4"切走期间仓库变更,切回看不到最新"的妥协。

watcher 注意点:
- Windows `recursive: true` 原生支持,但 `node_modules` 噪声大 → watch 时排除(或只 watch `.git/index` 的 mtime 变化 + git diff 重算)。
- 更稳的做法:**watch `.git/index`(staged 变化)+ 定时(每 3s)对工作区 `git status`**(untracked/modified 不写 index 也能捕获)。两路合一,避免递归 watch 整个仓库的性能坑。
- session 销毁时必须 `watcher.close()`(GitService 已有 clearTimers 模式,扩展即可)。

### 项目约束文案(写进 AGENTS.md §10 性能要求 + 软件定义书 §10)

新增条目:

> **面板切换延迟(2026-07-19 增补)**:
> - 有缓存时:Git / 文件 / 已打开 面板切换 **< 16ms**(1 帧,零感知)
> - 无缓存首拉:**< 300ms**(含 spawn git,Windows 上界)
> - 切走再切回:**必须命中缓存**,不得重新拉取
>
> 实现约束:面板级数据(如 GitPanel 的 status)在 renderer 端必须有**组件外缓存**(模块级 Map 或 store slice),组件卸载不丢数据;main 端的高频只读数据(如 git status)在 session 生命周期内**至少缓存 1 次**,且通过 watcher/event 主动失效。

### 工作量
- 层 1(组件外缓存):~40 行,30 分钟,**最高 ROI**
- 层 2(预取):~30 行(main 端 applyGitAvailability 顺带 getStatus + emit),20 分钟
- 层 3(watcher):~80 行 + 测试,1.5 小时,有 .git/index watch 的坑要注意

**建议先做层 1+2**(消除 95% 感知延迟),层 3 作为增强(消除"切走期间变更"的边角态)。

### 风险
- 缓存失效:watcher 未启用前,用户切走期间 `git commit` 了 → 切回看到的还是旧 status。层 1+2 不解决这个,层 3 解决。**短期可接受**(用户主动点刷新或重开 tab)。
- spawn git 慢:即使有缓存,后台刷新仍 spawn。可优化为 `git status --porcelain=v2` 只读 index(比读工作区快),但 Windows ConGit 启动开销是硬底线,~100ms 砍不掉。**缓存命中才是正解,不是优化 spawn**。

## 问题 3:highlight.js 没发挥(做"双层高亮")

### 根因
当前 DiffViewer 只用 highlight.js 的 **diff 语言**,它只产出 4 个 class(`hljs-addition/deletion/meta/comment`),**与方案 A 手写 CSS 行级着色没本质区别**。用户引入库的期望是"更好的高亮",而 highlight.js 真正的威力是**对 100+ 种编程语言做 token 级语法高亮**。

### 方案:双层高亮(代码语法 + diff 行色)

业界标准做法(GitHub / VS Code / GitLab):

1. **解析 diff 文件头**:`+++ b/src/foo.tsx` → 提取路径 → 推断语言(`.tsx` → `typescript`)
2. **对每行的"内容部分"(去掉 `+`/`-`/` ` 前缀)用该语言 hljs.highlight**:
   - `+const x = 1` → 去掉 `+` → `const x = 1` → TypeScript 高亮 → `const` 染关键字色,`1` 染数字色,`x` 默认色
3. **行级底色不变**:`.diff-line-add` 绿底 / `.diff-line-del` 红底 仍然套在最外层
4. **结果**:代码 token 色 + 行底色共存 = GitHub 同款视觉

### 语言推断(无需新依赖)

highlight.js 自带 `hljs.getLanguage('typescript')` 注册表,但**语言→hljs名映射**要自己维护一个小表(避免自动检测的歧义和性能开销):

```ts
const EXT_TO_HLJS: Record<string, string> = {
  ts: 'typescript', tsx: 'typescript', js: 'javascript', jsx: 'javascript',
  py: 'python', rb: 'ruby', go: 'go', rs: 'rust', java: 'java',
  c: 'c', cpp: 'cpp', cs: 'csharp', php: 'php',
  json: 'json', yaml: 'yaml', yml: 'yaml', toml: 'ini',
  html: 'xml', xml: 'xml', css: 'css', scss: 'scss',
  md: 'markdown', sh: 'bash', ps1: 'powershell', sql: 'sql',
};
```

diff 里可能有**多个文件**(multi-file diff),每个 `diff --git` 块重新解析 `+++ b/...` 切换语言。

### 包体积权衡(关键!)

按需 import 单语言:每个语言 ~2-5KB。若全部 import,highlight.js 回到 ~600KB。

**推荐:核心语言按需 + 其余回退 diff**:
- 默认注册 8 个高频语言:typescript / javascript / python / json / bash / yaml / markdown / xml(覆盖 90% 场景,~30KB)
- 未注册语言的文件 → 回退当前行为(纯行级着色,无 token 高亮)
- 未来若用户反馈某语言高频,加进注册表即可(零代码改动)

这比"全量 import 600KB"或"只支持 diff 语言"都合理。

### 按需 import 写法

```ts
import hljs from 'highlight.js/lib/core';
import diff from 'highlight.js/lib/languages/diff';
import typescript from 'highlight.js/lib/languages/typescript';
import javascript from 'highlight.js/lib/languages/javascript';
// ... 8 个核心语言
hljs.registerLanguage('diff', diff);
hljs.registerLanguage('typescript', typescript);
// ...
```

实测包体积:当前 +49KB(diff only),加 8 语言预计 **+80-100KB**(gzip ~+30KB)。可接受。

### 实施步骤

1. 扩展 `DiffViewer.tsx`:
   - 解析 diff 文本,按 `diff --git` / `+++ b/...` 分块
   - 每块推断语言(EXT_TO_HLJS 查表)
   - `buildRows` 时:对每行去掉前缀 → 用推断的语言 hljs.highlight(若语言未注册,回退 'diff' 语言)
2. 注册 8 个核心语言(模块级)
3. CSS:确保 `.hljs-keyword`/`.hljs-string`/`.hljs-number` 等 token 类**不被** `.diff-line-add` 的底色覆盖(当前我写了 `.diff-line-body .hljs-addition { background: transparent }` 是对的,但要保留 token 的 color)。需要引入一个 highlight.js 主题 CSS,或自己写一套匹配 Marina 7 主题的 token 色。

### CSS 主题适配(难点)

highlight.js 官方主题(github.css / github-dark.css)用固定色,不跟随 Marina 7 套主题。两个选择:

**选项 A:按 Marina 主题名动态加载 hljs 主题**(复杂):7 套主题 × 深浅 = 14 个 hljs 主题 CSS,运行时注入。工作量太大。

**选项 B:自己写一套 token 色,用 CSS 变量映射**(推荐):
```css
.diff-line-body .hljs-keyword { color: var(--color-info); }      /* const/import 等 → 蓝 */
.diff-line-body .hljs-string { color: var(--color-success); }    /* "..." → 绿 */
.diff-line-body .hljs-number { color: var(--color-warning); }    /* 123 → 橙 */
.diff-line-body .hljs-comment { color: var(--color-text-muted); }/* // ... → 灰 */
.diff-line-body .hljs-function,
.diff-line-body .hljs-title { color: var(--color-info); }
.diff-line-body .hljs-attr,
.diff-line-body .hljs-property { color: var(--color-text-primary); }
```
**一套 CSS 自动适配 7 套主题**(因为用的是主题变量)。这是正解。

### 工作量
~120 行(DiffViewer 重构 ~60 + 语言表 ~20 + CSS token 映射 ~40)+ 注册语言。1.5 小时。

### 风险
- diff 里语言切换的边界(一个 diff 含 .ts + .md 两个文件):解析 `+++ b/` 要正确分段。已在方案里考虑(按 `diff --git` 分块)。
- highlight.js 对 JSX/TSX 的支持有限(它不是 tree-sitter):`<Component prop={1}>` 可能着色不全。可接受(行级底色 + 部分着色已比纯文本好)。

## 问题 1:树形/平铺视图切换

### 根因
GitPanel 当前渲染:按 tone 分组(conflict/modified/.../untracked)→ 每组平铺 relativePath 列表。无目录树构建。

### 方案

**UI**:面板头部加一个 view-mode toggle(两个图标按钮:平铺 / 树形),本地 state,**持久化到 localStorage**(用户偏好,不进全局 settings —— 这是个 view 偏好不是产品设置)。

```tsx
const [viewMode, setViewMode] = useState<'flat' | 'tree'>(() =>
  localStorage.getItem('marina.git.viewMode') ?? 'flat');
```

**flat 模式**(当前):tone 分组 + 平铺列表。保留。

**tree 模式**:把所有变更文件(relativePath)构建成目录树,复用 FileListRow 的目录节点渲染(depth + 展开箭头 + folder icon)。

```tsx
interface TreeNode {
  name: string;
  path: string;          // 相对 repoRoot
  isDir: boolean;
  children?: TreeNode[];
  tone?: GitStatusTone;  // 叶子节点的变更状态
}
function buildTree(entries: Array<{path: string; tone: GitStatusTone}>): TreeNode[] { ... }
```

**树形视觉规则**(对齐 VS Code Source Control):
- 目录节点:folder icon + 名字。若该目录下有变更,**目录名也染色**(继承子节点最严重的 tone:conflict > modified > added > untracked > deleted)。
- 叶子节点:文件 icon + 名字 + tone badge(M/A/D 等),点击 open-diff。
- 目录可展开/收起(默认全展开,除非用户收起)。
- **平铺模式按 tone 分组的 header 在树形模式下隐藏**(树形已用目录结构组织,再按 tone 分组冗余)。或者:树形模式下顶部仍显示 tone 图例 + 计数,但不分组。

**rename 处理**:`R old → new` 在树形里按 `new` 路径放,hover 显示 old。平铺模式已有此处理。

### 工作量
- buildTree 纯函数 + 测试:~50 行,30 分钟
- tree 渲染(复用 FileListRow):~50 行
- view-mode toggle UI + localStorage:~20 行
- CSS 微调:~10 行

**1 小时**。

### 风险
- 目录嵌套深时(depth > 5)缩进挤压:可设 max-depth + 超深时退化为平铺显示叶子路径。实际仓库罕见。
- node_modules 等噪声目录:tree 模式下若 untracked 含 node_modules,整个目录树被污染。**解法**:tree 模式下,untracked 目录若子项 > 50,折叠显示"node_modules/(+ N untracked)"。或者依赖 `.gitignore`(git status 本身已尊重 .gitignore,不会列 node_modules 除非强制)。实际无问题。
