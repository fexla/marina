# 主工作区布局架构 —— 设计方案

> **状态**:✅ 已实现并完成自动验证(架构灵活 / UI 克制 / 受限双根文件导航)
> **日期**:2026-07-12
> **起因**:开发者要求主工作区架构"灵活到能支撑 JetBrains 系 IDE 式面板组合的地步,以便后续加面板(git diff、文件浏览等)",并询问内部面板应抽象为"窗口"还是"面板"。
> **方向澄清(2026-07-12)**:开发者明确——**要的是"架构层"的灵活性,不是"UI 层"暴露给用户自由拖拽**。架构内部抽象要足够通用,但面板的出现/位置/大小仍由程序驱动 + 产品规则决定。此澄清使需求与产品哲学不再冲突,落入合规技术问题。

---

## 0. TL;DR

1. **方向**:引入一套通用的"面板系统"抽象(Panel Registry + Layout Tree + Layout Host),让新增任何面板都不用改 `MainPane` 与 `SessionUiLayout` 的硬编码;当前仅生成终端主区 + 右 dock stack,且**不接受 renderer 提交布局树变更**——架构能表达组合,IPC 不开拖拽口子,哲学边界焊在产品规则层。
2. **设计原则一句话**:能力上限按 JetBrains 设计,默认行为按 Marina 哲学克制。
3. **命名**:绝不用"窗口"(Electron Window 已占用);通用抽象叫 **Panel**,内部区分 `PanelType` / `PanelInstance` / `LayoutNode` / `LayoutHost`;程序推送型用 `trigger: 'program-push'` 字段区分,不再单独造 Surface 词汇族。
4. **文件导航范围**:新增只读 `FileTreePanel`,仅允许浏览当前 session 的真实 `currentCwd` 与该 session 的 `MARINA_WORKSPACE`,选中文件后交给现有 Viewer 打开;不提供编辑、上传、删除、重命名或任意路径输入。
5. **合规性**:不引入 Project / Workspace 产品容器或内置编辑器;受限导航已写入软件定义书 §14.6 例外与 ADR-016。

---

## 1. 当前架构事实

### 1.1 产品定位基线与受限例外

- **§14.1**:`Marina = 终端管理器(本地 + SSH + 远程后端),不做文件管理`
- **§14.6**:文件管理是另一产品形态;"路径中心"指会话,不指文件浏览
- **§13.2 永远不做**:内置文件编辑器;Project / Workspace / Repo 容器层级
- **§2 哲学**:UI 临时(原则 1)、用户决策最少化(原则 3)、窗口平等(原则 4)

本方案在**架构层**提供灵活性,但不在交互层开放用户自由拖拽布局。开发者已明确批准一个窄范围产品例外:用户可在绑定当前 session 的 `FileTreePanel` 中只读浏览 `currentCwd` 与 `MARINA_WORKSPACE`,并指定文件交给现有 Viewer 打开。该例外不产生 Project / Workspace / Repo 容器,也不提供文件修改能力;实现前需同步修订 §14.6,不能继续声称它与现有绝对禁止文字天然相容。

### 1.2 当前实现的硬编码问题(为什么要改)

现在三个地方都把 `filePanel` 这个具体类型写死了,加第二个面板就要线性改这三处:

| 位置 | 现状 | 加新面板的代价 |
|---|---|---|
| `MainPane.tsx:196-202` | `<TerminalView/>` + 条件 `<FilePanel/>` 写在 JSX | 再塞组件,`terminal-area` 排版膨胀 |
| `SessionUiLayout` (types.ts:192) | `{ filePanel: { width, collapsed } }` | 改 schema,破坏序列化/迁移 |
| `updateUiLayout` (session-manager.ts:1014) | `filePanel` 专属校验,硬编码 `[280,900]` | 每种面板复制一份校验 |

### 1.3 当前 file-panel 的性质(不变)

`file-panel` 是**终端里跑的程序经 `MARINA_SERVICE` HTTP API 主动推送、per-session 绑定、被动展示**的区域(`FilePanel.tsx`:`files.length === 0 → return null`;`file-panel-service.ts` 头注)。新架构必须保留这个性质,不把它误提升成"用户主动管理的 UI 元素"。

新增的文件导航不能塞进 `FilePanel`:它应是独立的 `FileTreePanel`(`trigger: 'product-rule'`),负责列目录和选择文件;选择结果再交给现有 Viewer/FilePanel 内容渲染能力。这样程序推送状态与用户导航状态不会混成一个状态机。

### 1.4 FileTreePanel 的产品边界(已确认)

`FileTreePanel` 绑定 active session,只暴露两个逻辑根:

1. `session-cwd`:当前 session 的真实 `currentCwd`,随 cwd 更新而更新;
2. `managed-workspace`:该 session 的 `MARINA_WORKSPACE` 受管目录。

硬边界:
- 只读列目录、查看基础元数据、选择文件并在现有 Viewer 中打开;
- 不提供编辑、创建、重命名、删除、上传、下载或拖放文件操作;
- 不允许文本输入任意路径,也不允许从树中跳到两个根之外;
- 路径校验必须基于 `realpath` 后的根包含关系,符号链接 / junction 不得绕过根边界;
- 两个根是 session 能力,不是可收藏、命名或持久化的 Project / Workspace 容器;
- 本地 session 可直接通过后端文件 API 浏览;远程后端 session 由远程后端执行同样的受限 API;传统 SSH session 不因此引入 SFTP,没有可用文件 API 时面板明确显示“不支持”。

## 2. 设计原则:能力上限高,默认行为克制

数据结构和抽象按 JetBrains 的上限设计,但 V1 的可见行为只用到它最浅的一层,且由产品规则生成、不接受用户编辑。这样既满足"架构灵活"的诉求,又把哲学边界焊在"产品规则"层,而不是靠抽象缺失来保证——后者反而会在未来某天被"顺手开了"。

---

## 3. 三个核心抽象

### 3.1 Panel Type Registry(面板类型注册表)

每种面板是一个自描述的 `PanelType`,注册进全局表。新面板 = 加一项注册,不改 `MainPane`。

```ts
interface PanelType {
  typeId: string;                              // 'file-tree' | 'file-panel' | 'git-diff-push' ...
  Component: React.FC<PanelInstanceProps>;
  trigger: 'program-push' | 'product-rule';    // 谁让它出现
  defaultDock: DockSide;                       // 'right' | 'bottom' | 'left'(产品定死)
  defaultState: PanelLayoutState;              // 默认宽/折叠/可见
  validateState?: (s: unknown) => PanelLayoutState;  // 替代 updateUiLayout 的硬编码校验
}
```

`file-panel` 的"程序推送 + per-session"体现为 `trigger: 'program-push'`；首个 `file-tree` 是 `trigger: 'product-rule'`，但只允许 session 的 cwd/workspace 双根。未来 `agent-todo`(claude code 推 TODO)、`git-diff-push`(终端里的 git 程序推 diff 预览)都是前者同型，各自注册一项。

### 3.2 Layout Tree(布局树,内部数据结构)

这是"JetBrains 级灵活性"的承载点 —— 一个可递归的布局树:

```ts
type LayoutNode =
  | { kind: 'split'; dir: 'horizontal' | 'vertical'; children: LayoutNode[]; ratios: number[] }
  | { kind: 'stack'; children: LayoutNode[]; defaultActivePanelId: string }
  | { kind: 'leaf';  panel: PanelInstance };
```

理论上能表达 JetBrains 任意嵌套。**但 V1 产品规则只生成这种浅树:**

```
split(horizontal,
  [ leaf(TerminalView),           // 主区,产品钉死
    stack(right,
      [ leaf(file-tree),          // 产品规则：cwd/workspace 双根只读导航
        leaf(file-panel) ]) ])    // 程序推送的已打开文件预览
```

**关键纪律:V1 不接受 renderer 提交 `tree` 变更。** tree 由 main 端按产品规则生成/维护,renderer 只能改 `docks.right` 的几何态(宽/折叠)。宽度属于 dock 而不是其中的页面，切换文件树/已打开文件不会改变布局。这一条直接焊死"用户自由拖拽"的可能 —— 架构能拖,但 IPC 不开这个口子。

### 3.3 Layout Host(布局容器组件)

一个通用容器,读 `tree` + `registry` 递归渲染,负责所有通用 chrome(分割条、tab 条、dock 边框)。**面板组件只写内容,不写位置。** `FilePanel` 现在自带的 `resize-handle` / 折叠条下沉到 LayoutHost,所有面板共享。

## 4. 数据模型演进(最小破坏)

```ts
// before
interface SessionUiLayout {
  filePanel: { width: number; collapsed: boolean };
}

// after
interface SessionUiLayout {
  tree: LayoutNode;                              // 布局树(产品规则维护,V1 不接受 renderer 改)
  docks: Record<string, DockLayoutState>;        // dock 几何态,当前仅 right
}
interface DockLayoutState { width: number; collapsed: boolean }
```

迁移:
- 旧 `uiLayout.filePanel` 的宽/折叠态 → `uiLayout.docks.right`；不为 file-tree/file-panel 分别存宽度
- `updateUiLayout` 的 `[280,900]` 校验由 main 端 Dock registry 驱动
- `session-manager.test.ts` 的 uiLayout 断言更新(预期内破坏,迁移路径写明)
- `uiLayout.version` 升为 2；session UI 不跨应用重启持久化，因此无需磁盘迁移

## 5. 组件分层

```
App
└─ MainPane
   └─ LayoutHost                    ← 新:读 tree + registry,递归渲染
      ├─ leaf: TerminalView          (主区,产品规则钉死,不进 registry)
      └─ stack(right):
         ├─ leaf: FileTreePanel      (registry['file-tree'],cwd/workspace 双根只读)
         ├─ leaf: FilePanel          (registry['file-panel'],程序推送的已打开文件)
         └─ leaf: (未来)GitDiffPush / AgentTodo / ...
```

`TerminalView` 是特殊 leaf —— 它不是注册的 PanelType(它是终端管理器的本体),由产品规则直接钉在主区。Dock 区里的才是注册面板。这个区分很重要:防止日后有人把终端也降格成一个"可关闭的面板"。

## 6. 迁移路径(不破坏现有行为)

1. **抽 PanelType registry**,把 `FilePanel` 注册进去(`typeId: 'file-panel'`,`trigger: 'program-push'`,`defaultDock: 'right'`,把 `[280,900]` 校验搬进 `validateState`)。行为不变。
2. **MainPane** 的 `<FilePanel/>` 替换为 `<LayoutHost>`,LayoutHost 按 V1 浅树渲染出等价结果。行为不变。
3. **SessionUiLayout 泛化**:`filePanel` 字段 → `docks.right`,加 `tree` 与版本演进。`updateUiLayout` 改读 Dock registry 校验，切换页面保持同一 dock 宽度。
4. **更新测试**:`session-manager.test.ts` uiLayout 断言;`remote-loopback.test.ts` 的 `SESSION_UPDATE_UI_LAYOUT` 用例。
5. **通用 chrome 下沉**:`FilePanel` 的 resize-handle / 折叠条迁到 LayoutHost,`FilePanel` 只剩内容。
6. **实现首个新面板 `FileTreePanel`**:增加后端受限目录 API,同时暴露 `currentCwd` 与 `MARINA_WORKSPACE` 两个根;选择文件复用现有 Viewer。每次请求做 owner + `realpath` 根包含校验，传统 SSH session 无文件 API 时显示不支持,不引入 SFTP。
7. 每步独立 commit,`npm run typecheck && npm test && npm run lint` 通过。

整个过程在 feature branch(`checkpoint-4` 的勘误分支或新开 `feat/panel-system`,视当前所处检查点而定),不动 main;file-panel / MainPane 是已通过代码,重构走分支 + 自测(AGENTS.md §7 封箱规则在勘误/强需求时解冻)。

## 8. 命名定论

- **绝不用"窗口"**:Electron Window 已是一等公民(§8.4 `ownerWindowId`、`WindowManager`),一个 Electron 窗口里再套"窗口"术语当场崩溃。
- 通用抽象叫 **Panel**(面板)。内部区分:
  - `PanelType` —— 注册的面板类型(描述符)
  - `PanelInstance` —— 某个 session 里某类型的实例(带 state)
  - `LayoutNode` / `LayoutHost` —— 布局树 / 容器
- 程序推送型面板用 `PanelType.trigger: 'program-push'` 字段区分,**不**单独造 Surface 词汇族,避免概念膨胀。上一版提的 `SessionSurface` 降级为该 trigger 值的语义说明,不作类型名。

## 10. ADR-016 草案:面板系统架构(Panel System)

> 状态:已采纳并写入 软件定义书.md §10.2（ADR-016）

**背景**:主工作区当前针对 `filePanel` 单类型硬编码(`MainPane` JSX、`SessionUiLayout` schema、`updateUiLayout` 校验三处)。未来需容纳多种程序推送型面板(git diff 预览、agent todo 等),需一个可扩展架构。

**决策**:
- 引入 Panel Type Registry + Layout Tree + Layout Host 三层抽象。
- Layout Tree 数据结构按 JetBrains 上限设计(split/stack/leaf 递归),但当前产品规则只生成浅树(主区 TerminalView + 右 dock 面板组)。
- `SessionUiLayout` 泛化为 `{ tree, docks: Record<dockId, DockLayoutState> }`，当前仅 `right`，schema 版本为 2。
- **renderer 不可提交 `tree` 变更** —— 布局树由 main 端产品规则维护，IPC 只开放 `docks.right` 几何态(宽/折叠)。此条焊死"用户自由拖拽布局"的可能，也保证页面切换不改变 dock 宽度。
- `TerminalView` 不进 registry,由产品规则钉为主区 leaf,防止终端被降格为可关闭面板。
- 新增独立 `FileTreePanel`,只读浏览 active session 的 `currentCwd` 与 `MARINA_WORKSPACE`;它不改变程序推送型 `FilePanel` 的状态机。
- `FileTreePanel` 不提供任意路径输入或文件修改,并用 canonical `realpath` 根包含校验阻止符号链接 / junction 越界。
- 用户主动 Git 仓库浏览不在本次决策内;架构可承载,产品暂不开。
- 上述文件导航是 §14.6 的受限例外,实现前先更新软件定义书,避免规范与代码互相矛盾。

**后果**:
- 正面:加新面板 = 注册一项 PanelType,三处硬编码消除;架构留有未来"开放局部拖拽 / 加下面板"的扩展口,无需重写。
- 负面:引入 LayoutHost 递归渲染与布局树序列化的少量复杂度;`SessionUiLayout` schema 变更需版本迁移;`session-manager.test.ts` uiLayout 断言需更新。
- 不变:file-panel 的程序推送 + per-session 性质完全保留;哲学边界不被动摇。
